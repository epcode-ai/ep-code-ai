## 1. 测试范围

### 1.1 In Scope

- ✅ REQ-001 ~ REQ-006 全部 6 个需求
- ✅ 5 类假期的业务规则（年假/病假/事假/婚假/丧假）
- ✅ 审批状态机 4 个转换路径（通过/驳回/问询/取消）
- ✅ 余额扣减规则（跨年、半天、病假超额）
- ✅ 权限（员工/主管/HR 三角色）
- ✅ 非功能：性能、安全、兼容性

### 1.2 Out of Scope

- ❌ SSO 系统本身的测试（外部依赖）
- ❌ 邮件服务器稳定性（外部依赖）
- ❌ 多级审批（v2 才做）
- ❌ 移动端适配（v2）

| 类型 | 占比 | 负责 | 方式 |
|------|------|------|------|

## 3. 风险评估

| 风险 | 可能性 | 影响 | 优先级 | 缓解 |
|------|-------|------|-------|------|
| SSO 集成失败 | 中 | 高 | P0 | 提前 1 周接入测试环境 |
| 跨年请假规则复杂 | 高 | 中 | P0 | 专门 5 条用例覆盖 + 代码评审 |
| 并发审批竞态 | 中 | 高 | P0 | 专门并发测试 + 代码加锁 |
| 余额计算精度 | 中 | 中 | P1 | 用 decimal 类型 + 精度测试 |
| 邮件通知延迟 | 中 | 低 | P2 | 异步通知，不影响主流程 |
| 大数据量导出慢 | 高 | 低 | P2 | 限制导出数量 + 流式响应 |

## 4. 用例覆盖策略

### 4.1 覆盖度目标

| 维度 | 目标 |
|------|------|
| 核心业务流程 | 100% |
| 异常场景 | ≥ 80% |
| 边界值 | ≥ 90% |
| 状态转换 | 100%（4 条路径 × N 状态） |
| 权限矩阵 | 100% |

### 4.2 用例设计方法

| 方法 | 适用场景 |
|------|---------|
| 等价类划分 | 假期类型、天数、状态 |
| 边界值 | days (0, 0.5, 1, 30, 30.5), reason 长度 (9, 10, 500, 501) |
| 决策表 | 业务规则里的假期额度、审批规则 |
| 状态转换 | 请假单状态机 |
| 场景法 | 端到端用户场景（员工 → 主管 → HR） |
| 错误推测 | 基于经验,如"跨年冻结期"、"主管离职后" |

### 5.2 基础数据准备脚本

```bash
# 由开发提供，测试运行
./scripts/seed-test-data.sh
```

脚本内容:
- 创建 10 个测试账号
- 创建 2026 年假期余额记录
- 创建 50 条历史请假单（各种状态）
- 清理旧数据（可选）

### 5.3 数据清理

每轮测试后执行:
```sql

DELETE FROM leave WHERE reason LIKE 'test_%';
DELETE FROM leave_audit WHERE leave_id IN (...);
```

## 8. 自动化策略

### 8.1 自动化范围

- ✅ 接口测试 100% 自动化（Tavern）
- ✅ 核心 happy path 端到端自动化（Playwright）
- ⚠️ 边界/异常场景：部分自动化
- ❌ 视觉回归、UI 细节：手工

### 8.2 自动化运行

- 每晚跑全量，通过率 ≥ 95% 继续
- PR merge 时跑核心冒烟
- 上线前跑完整回归
