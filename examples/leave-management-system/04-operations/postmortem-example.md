## 5. 根因分析（5 Why）

**问题**: 团队日历接口导致 DB 连接池饱和

**Why 1**: 为什么 DB 连接池饱和?
→ 单个请求触发了大量（20+）的 SQL 查询

**Why 2**: 为什么会触发这么多查询?
→ `getTeamCalendar` 使用了 for 循环,每个团队成员单独查

**Why 3**: 为什么这么写?
→ 开发时没考虑到团队规模（20-30 人）的批量访问

**Why 4**: 为什么代码评审没发现?
→ 评审者关注业务逻辑正确性,对"N+1"这类性能陷阱不敏感

**Why 5**: 为什么测试没发现?
→ 测试用例只有"单人查询",没有"多人并发查询同一团队"的场景

**根本原因**:
1. **技术层面**: N+1 查询模式
2. **流程层面**: 代码评审 checklist 缺"性能反模式"检查项
3. **测试层面**: 没有并发场景测试
4. **文化层面**: 对"上线后流量特征"预估不足

## 6. 贡献因素

- **灰度节奏**: 4/25 周五凌晨上线 + 4/26 周一早高峰 → 全量压力集中暴发
- **监控缺口**: 没有 N+1 检测告警（如"单请求 SQL 次数 > 10"）
- **测试环境**: 测试环境只有 5 个团队成员,生产团队有 20+,压力差异大

## 11. 教训

### 普适教训

1. **"在测试环境没问题"不代表生产没问题** — 数据规模、并发模式是关键差异
2. **N+1 是老经典问题但依然高频** — 需要工具+流程+培训三管齐下
3. **回滚优于修复** — SEV-1/2 时先恢复业务，根因追后处理
4. **Blameless 文化让团队敢报问题** — 本次 junior 开发直接承认代码问题,没有遮掩

### 本项目特有

- 请假管理这类"每天固定时段集中使用"的系统,对并发更敏感
- 上线时机的选择要考虑"使用高峰"
