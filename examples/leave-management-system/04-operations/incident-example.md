## 4. 详细经过

### 4.1 触发

昨晚（4/25）凌晨上线了 v1.0.1 小版本,主要是修复 UI 显示和加了 HR 请求的"团队假期日历"接口。上线后一切正常，通过了冒烟。

今早 09:00 开始，灰度范围内的 HR 同学陆续到岗开始使用新功能。9:15 时 HR 小组同时打开"团队日历"，触发告警。

### 4.2 诊断

- **9:22**: Grafana 显示 DB 连接池 48/50,几乎打满
- **9:25**: 查 `pg_stat_statements` 发现一条 SQL 被调用 3000+ 次/分钟：
  ```sql
  SELECT * FROM leave WHERE user_id = $1 AND start_date BETWEEN $2 AND $3;
  ```
- **9:28**: 赵工查看代码发现,`getTeamCalendar` 用了循环:
  ```typescript
  // 问题代码
  for (const userId of teamMemberIds) {
    const leaves = await prisma.leave.findMany({ where: { userId } })
    // 组装日历
  }
  ```
  每次团队有 20 个成员，就是 20 次查询。10 个 HR 同时打开就是 200 次。

### 4.3 处置

决策: **回滚** v1.0.1 → v1.0.0

理由:
- 故障直接关联新功能
- 修代码需要至少 30 分钟（影响 SLA）
- 团队日历是 P1 功能,推迟几天可接受

### 4.4 验证

- 应用 rollout undo 后，各 Pod 重启
- API 恢复正常 P99 < 300ms
- DB 连接池水位降回 5-10
- HR 同学反馈"可以正常用了"

## 7. 初步根因（热分析）

- **直接原因**: `getTeamCalendar` 接口 N+1 查询,流量一上就打满 DB 连接池
- **为什么没在测试阶段发现**: 测试时每个 HR 单独操作,N+1 的指数放大没体现
- **为什么代码评审没发现**: 评审者关注业务逻辑,对 N+1 模式不敏感
- **为什么冒烟测试没发现**: 冒烟用例没模拟"10 个 HR 同时操作"
