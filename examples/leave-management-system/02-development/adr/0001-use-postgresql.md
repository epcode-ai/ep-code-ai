## 背景与上下文

员工请假管理系统 v1.0 需要选一个主数据库。候选方案：

- **MySQL 8**: 公司多数系统在用
- **PostgreSQL 15**: 本组部分同学有经验
- **MongoDB 7**: 文档数据库

### 要解决的问题

- 存储请假、余额、审计等结构化数据
- 支持复杂查询（HR 统计场景）
- 保证事务一致性（审批时同时改状态和余额）

### 约束

- 预算：紧，首选免费开源
- 运维能力：公司 DBA 熟悉 MySQL 和 PG，不熟悉 Mongo
- 规模：300 用户，年请假 3000 条，5 年后 ~1.5 万条（不大）
- 合规：数据留存 3 年

## 决策

**使用 PostgreSQL 15**。

## 理由

### 1. 强事务支持

审批时要同时更新 `Leave` 和 `LeaveBalance`，必须事务。PG 和 MySQL 都支持，但 PG 默认隔离级别更严格（READ COMMITTED），MySQL 默认是 REPEATABLE READ，PG 的事务语义更直接。

### 2. 丰富的数据类型

- 原生支持 `decimal` 高精度（余额计算）
- 原生支持 `enum`（状态、假期类型）
- 原生支持 `jsonb`（留给 v2 多级审批审批链扩展）
- 原生支持数组（如"多个直接下属"）

### 3. 复杂查询性能好

HR 导出场景要做复杂聚合（按部门、类型、月份分组）。PG 的查询优化器和窗口函数比 MySQL 更成熟。

### 4. 团队经验

- 赵工之前在上家公司用 PG 5 年
- 其他 2 位后端有 PG 开发经验
- 运维有 DBA 支持

### 5. 全文搜索

原生支持中文全文搜索（配合 jieba 扩展），方便后续"根据原因搜索历史"功能。

## 后果

### 正面

- ✅ 事务和数据类型匹配业务需求
- ✅ 可以减少一个 ElasticSearch 依赖（用 PG FTS 够用）
- ✅ 团队现有知识可复用

### 负面

- ❌ 公司 DBA 对 PG 运维工具（如 pg_stat_statements）没 MySQL 熟悉
  - 缓解：列入 DBA 培训计划，参考公司其他 PG 项目的 runbook
- ❌ PG 的主从复制配置比 MySQL 复杂
  - 缓解：v1.0 先用单主，不上从库；v2 再加

## 备选方案（未选）

### 方案 A：MySQL 8

**优点**:
- 公司标配，运维成熟
- 社区资源丰富

**为什么没选**:
- 枚举类型实现差（用 VARCHAR + CHECK 约束，扩展性不好）
- 复杂聚合查询比 PG 慢
- 对 decimal 支持 MySQL 也有，不是决定性差异

### 方案 B：MongoDB

**优点**:
- schema flexible
- 文档模型直观

**为什么没选**:
- 本业务是强事务（审批同时改多表），Mongo 4.0+ 虽然支持事务但性能和 RDBMS 差距大
- 团队不熟悉，运维风险大
- 聚合查询（HR 统计）写起来复杂
