# 代码翻译

## 用途

老代码迁移到新语言时，先让 AI 出初稿，再人工审核。

## 使用场景

- 老 Java 代码迁到 Kotlin
- Python 2 → 3
- JavaScript 迁到 TypeScript
- Ruby → Go（微服务拆分时）

## Prompt 本体

```markdown
把这段 [源语言] 代码翻译成 [目标语言]。

## 要求
1. **保持功能完全一致**（行为不变）
2. 使用目标语言的**惯用法**（idioms），不是逐字翻译
3. 补充目标语言所需的类型声明
4. 异常处理按目标语言习惯
5. 命名按目标语言约定（Python snake_case / Java camelCase / ...）
6. 依赖库用目标语言主流选择

## 源代码
```[源语言]
[粘贴]
```

## 上下文
- 源语言版本:
- 目标语言版本:
- 目标运行环境:
- 已有的依赖（目标语言）:
- 团队代码风格约定:

## 输出

### 1. 翻译后的代码
```[目标语言]
[完整代码]
```

### 2. 关键差异说明
| 源语言写法 | 目标语言写法 | 为什么这样翻 |
|----------|------------|------------|
| 源代码片段 | 目标代码片段 | 理由 |

### 3. 潜在的语义差异
警告！以下地方源代码和目标代码行为可能不完全一致：
- 差异点 1: [描述 + 风险]

### 4. 目标语言特有的陷阱
- 陷阱 1: [如 Go 的 nil map 不能赋值、Python 可变默认参数]

### 5. 建议的测试重点
- 点 1: [为什么要测这里]
- 点 2: ...

### 6. 依赖映射
| 源依赖 | 目标依赖 | 说明 |
|-------|---------|------|

## 约束
- 不要做无关的"顺手重构"
- 不懂的地方用注释标出"⚠️ AI 不确定，需人工确认"
```

## 使用示例

**输入**: 50 行 Python 数据处理代码 → Go

**输出**: 地道的 Go 代码 + 关键差异说明（如 list comprehension → for loop）
