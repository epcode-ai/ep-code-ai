# 代码重构

## 用途

给出保持功能不变的重构方案，降低复杂度、提升可读性。

## 使用场景

- 遇到"又长又乱"的老代码
- 准备给老代码加新功能前
- 代码评审时 reviewer 觉得结构不好

## Prompt 本体

```markdown
你是一位资深软件架构师。帮我重构以下代码。

## 重构目标
- [ ] 降低复杂度（函数/文件过长、嵌套过深）
- [ ] 提升可读性（命名、结构）
- [ ] 提取可复用逻辑
- [ ] 职责分离（SRP）
- [ ] 便于测试
- [ ] 保持原有行为不变（最重要）

## 约束
1. **不改变公共接口**（签名、返回值）
2. **不引入新依赖**（除非明确说明）
3. **保持原有测试全部通过**
4. **改动要渐进式**（不要一次颠覆）

## 输出

### 1. 重构后的代码
```语言
[完整代码]
```

### 2. 改动清单
逐项说明：
- 改了什么（before → after）
- 为什么改
- 对应哪个重构原则（EXTRACT_METHOD / RENAME / ...）

### 3. 验证步骤
- 运行哪些测试
- 手工验证哪些场景

### 4. 可能的风险
- 潜在的行为变化点（需要人工 review）
- 可能影响的其他模块

### 5. 后续建议（超出本次范围）
- 如果还要继续改进，下一步做什么

## 原代码
```语言
[粘贴]
```

## 上下文
- 这段代码的职责:
- 调用方:
- 被调用方:
- 现有测试覆盖度:
```

## 使用示例

**输入**: 一个 300 行的 `UserService.processOrder` 函数

**输出**: 拆成 5 个小函数 + 提取 helper + 加类型,对公共接口零改动
