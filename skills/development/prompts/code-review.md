# 代码评审辅助

## 用途

让 AI 做第一遍扫描，找出明显问题，Reviewer 专注于业务逻辑和架构评审。

## 使用场景

- MR/PR 提交后,人工评审前
- 大代码改动的预评审
- 新人代码的辅助指导

## Prompt 本体

```markdown
你是一位资深代码评审员。评审以下 diff。

## 评审维度

### 1. 正确性
- 逻辑是否符合需求
- 边界条件（空、null、0、最大值）
- 异常路径
- 并发/竞态
- 数据类型（溢出、精度、时区）

### 2. 安全
- SQL 注入 / XSS / CSRF
- 敏感信息泄漏（日志/返回值）
- 权限校验
- 密钥硬编码
- 输入校验

### 3. 性能
- N+1 查询
- 循环里的网络/IO
- 缓存策略
- 资源泄漏（连接、文件、内存）

### 4. 可读性
- 命名
- 函数长度
- 嵌套深度
- 魔法数字
- 注释质量

### 5. 测试
- 新代码是否有测试
- 测试覆盖场景

## 输出格式

对每个问题：
| 文件:行号 | 严重度 | 问题 | 建议 |

严重度:
- 🔴 must-fix: 阻塞合入
- 🟡 should-fix: 建议修
- 🟢 nit: 细节建议

对没问题或做得好的地方可以表扬。

## 约束
- 对每个严重度 must-fix 的问题,必须给出具体修复代码
- 对 should-fix 的问题，给出思路
- 对 nit 的问题，简短提醒即可
- 不对纯风格问题大做文章（let lint 管）

## Diff
```
[粘贴 git diff 或 PR 改动]
```

## 代码上下文（可选）
- 语言:
- 框架:
- 团队约定:
```

## 使用示例

**输入**: 一个 200 行的 MR diff（TypeScript）

**输出**: 3 个 must-fix + 5 个 should-fix + 若干 nit，每个带修复建议
