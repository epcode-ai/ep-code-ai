# SQL 优化

## 用途

分析慢查询的 EXPLAIN 结果，给出分层优化方案（从小改动到大改造）。

## 使用场景

- 慢查询告警
- 性能测试发现的瓶颈
- 大表查询前的预审核

## Prompt 本体

```markdown
你是一位资深 DBA。分析以下慢查询，给出优化方案。

## SQL
```sql
[粘贴 SQL]
```

## EXPLAIN 输出
```
[粘贴 EXPLAIN / EXPLAIN ANALYZE 的完整输出]
```

## 表信息
- 表 A 大小: [行数 + 存储大小]
- 表 B 大小: ...
- 现有索引:
  - idx_xxx: 列名
  - idx_yyy: 列名
- 主键: ...

## 查询信息
- 调用频率: [QPS]
- 业务场景: [描述]
- 可接受延迟: [X ms]

## 当前问题
- 实测延迟: [X ms]（目标: Y ms）
- 是否命中索引: [查看 EXPLAIN]
- 扫描行数: [数字]

## 输出

### 1. 慢的根本原因
[1-3 句话解释]

### 2. 优化方案（按改动成本从小到大）

**方案 A: [名称]**
- 改动: [具体怎么改]
- 预期效果: [提升百分比]
- 风险: [低/中/高]
- 成本: [开发 X 小时]

**方案 B: [名称]**
- ...

**方案 C: [名称]**
- ...

### 3. 推荐方案及理由

选择方案 [X]，因为:
- 性价比最高
- 风险可控
- 符合业务场景

### 4. 验证方法

- 改动前/后的 EXPLAIN 对比
- 压测验证
- 监控观察

### 5. 需要注意的副作用

- 加索引的副作用: 写入变慢、空间占用
- 查询改写的副作用: [如有]
- 数据库配置改变的副作用: [如有]
```

## 使用示例

**输入**: 一个 3 表 JOIN + 日期范围查询，1 亿行数据，P99=8 秒

**输出**: 3 个优化方案（加索引、查询改写、分库分表）+ 推荐方案
