# Bug 汇总分析

## 用途

对一段时间的 Bug 数据做统计分析，找出高频根因和改进方向。

## 使用场景

- 版本 Bug 回顾
- 季度质量复盘
- 对比不同模块的质量

## Prompt 本体

```markdown
分析以下 Bug 数据,找出规律和改进方向。

## 数据范围
- 时间段:
- 项目/模块:

## Bug 原始数据
```
[粘贴 Bug 列表，字段：ID/标题/严重度/模块/创建人/修复人/状态/创建时间/关闭时间/根因/修复方式]
```

## 任务

### 1. 基础统计
- 总数
- 按严重度分布（P0/P1/P2/P3）
- 按模块分布
- 按状态分布（已关闭/进行中/重开）
- 平均修复时长（按严重度）

### 2. 趋势分析
- 时间维度的 Bug 数变化
- 严重 Bug 的时间分布
- 修复效率趋势

### 3. 模块热点
- Bug 最多的 top 5 模块
- 严重 Bug 最多的 top 5 模块
- 每个模块的 Bug 密度（和代码量对比）

### 4. 根因模式（核心）
- 如果数据里有根因字段，按根因分类
- 如果没有，根据标题/描述推测

常见根因类型：
- 需求理解错误
- 边界条件遗漏
- 并发问题
- 性能问题
- 第三方依赖
- 环境/配置
- 回归（老 Bug 复现）
- 其他

### 5. Bug 重开率分析
- 重开的 Bug 有什么共性
- 重开的根本原因

### 6. 逃逸 Bug 分析（线上发现的）
- 线上 Bug 占比
- 为什么测试阶段没发现

### 7. 改进建议
基于发现的模式,给出具体改进项:
- 技术层面: [如某模块加测试覆盖]
- 流程层面: [如代码评审加 XX 检查]
- 文档层面: [如补充 XX 规范]

## 输出

### 执行摘要
（1-2 段话的核心结论）

### 详细数据
（图表 + 表格）

### 关键发现
top 3-5 个值得管理层关注的点

### 改进行动项
SMART 原则:具体 + 可衡量 + 有责任人 + 有截止日期
```

## 使用示例

**输入**: 季度 150 个 Bug 的数据

**输出**: 结构化分析,发现订单模块 Bug 最多、并发问题是高频根因、建议加 xx 测试
