# 接口变更影响分析

## 用途

接口变更后，快速识别哪些是破坏性变更、哪些用例需要更新/回归。

## 使用场景

- 开发改了接口,测试要评估影响
- 版本升级前的接口审查
- 客户端适配新接口

```markdown

## 旧版接口
```
[粘贴 v1 的接口文档]
```

## 新版接口
```
[粘贴 v2 的接口文档]
```

- 现有测试用例: [数量]
- 现有自动化脚本: [数量]
- 现有的 Mock/测试桩: [数量]
- 客户端: [列表：前端/iOS/Android/第三方]

## 任务

### 1. 变更清单
识别所有变更，分类:
- 🔴 **破坏性变更**（Breaking）: 老客户端/老测试会失败
- 🟡 **行为变更**（Behavior）: 不影响签名但语义变了
- 🟢 **新增**（Added）: 新字段/新接口/新错误码
- ⚫ **废弃**（Deprecated）: 标记废弃但还能用
- ❌ **移除**（Removed）: 彻底删除

### 2. 详细影响（破坏性变更）

|------|---------|---------|---------|

- 预估测试工期: [天]

### 5. 兼容性建议
- 是否需要灰度: [是/否 + 理由]
```
