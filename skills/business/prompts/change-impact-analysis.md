# 变更影响评估

## 用途

当产品/业务提出变更请求（CR）时，快速评估影响，避免漏掉影响面。

## 使用场景

- 变更评审会前
- 紧急变更决策时
- 版本冻结期是否放行某个变更

## Prompt 本体

```markdown
你是一位资深系统分析师。帮我评估这个变更的影响。

## 变更描述
[粘贴 CR 的变更内容 + 原因 + 期望效果]

## 当前系统上下文
- 业务模块: [简述]
- 涉及的服务: [列表]
- 当前版本: v[X]
- 是否已在测试阶段: 是/否

## 分析维度

### 1. 功能影响
- 涉及哪些功能点
- 功能逻辑需要修改的部分

### 2. 数据影响
- 涉及哪些数据模型
- 是否需要 DB Migration
- 存量数据处理方案

### 3. 接口影响
- 涉及哪些 API
- 是否是破坏性变更
- 对客户端的影响

### 4. 性能影响
- 预期性能变化
- 可能的瓶颈

### 5. 安全影响
- 可能引入的风险
- 需要加强的校验

### 6. 用户体验影响
- 用户能感知到什么变化
- 是否需要培训/通知用户

### 7. 测试影响
- 需要重新覆盖的场景
- 预估测试工期
- 是否影响当前测试进度

### 8. 文档影响
- 需要更新的文档清单

### 9. 下游影响
- 依赖此功能的其他团队/系统

### 10. 时间影响
- 预估开发工期
- 是否影响发布计划

## 输出格式

| 维度 | 影响 | 严重度（高/中/低） | 具体说明 |

## 总结
- 总体影响评级: 高/中/低
- 是否建议接受: 是/否/有条件接受
- 如有条件,条件清单
- 风险提示
```

## 使用示例

**输入**: "把批量导出上限从 1 万条改成 5 万条"

**输出**: 涉及性能、测试、接口文档等多个维度的结构化评估
