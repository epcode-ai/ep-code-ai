# PRD 可测性分析

## 用途

在 PRD 评审阶段，快速扫描全文,找出阻碍测试的问题。

## 使用场景

- 需求评审会前 1 天
- PRD 定稿前的最后检查
- 测试拿到 PRD 准备写测试策略前

## Prompt 本体

```markdown
你是一位资深测试架构师。请分析以下 PRD，从"可测性"角度找出问题。

## 检查维度
1. 哪些需求点**缺少验收标准**（无 Given-When-Then）
2. 哪些描述**过于模糊**（如"用户体验好"、"响应快"）
3. 哪些**边界条件未明确**（数值范围、长度限制）
4. 哪些**异常场景缺失**（网络异常、并发、超时）
5. 哪些**数据规则不清**（精度、格式、必填）
6. 哪些**权限规则未说明**（谁能做什么）
7. 哪些**性能指标未量化**（QPS、延迟、容量）
8. 哪些**兼容性要求未明确**（浏览器、系统、版本）

## 输出格式
Markdown 表格,字段:
| 问题编号 | 所在章节 | 问题类型 | 具体问题 | 建议补充 |

## 附加输出
1. 整体可测性评分（1-10 分，10 为最可测）
2. 是否建议通过评审（通过 / 有条件通过 / 不建议通过）
3. 关键问题清单（top 3 必须解决）

## PRD 内容
[粘贴 PRD 或附上链接内容]
```

## 使用示例

**输入**: 一份电商"批量导出订单"功能的 PRD

**输出示例**:
```
| # | 章节 | 问题类型 | 具体问题 | 建议补充 |
|---|------|---------|---------|---------|
| 1 | 3.1 | 验收缺失 | "批量导出"无 AC | 补充 Given-When-Then |
| 2 | 4.1 | 性能模糊 | "响应要快" | 量化: P99 ≤ 500ms |
| 3 | 3.2 | 边界未定 | "大批量" | 明确上限: 10000 条 |

整体可测性: 6/10
建议: 有条件通过（修复上述 3 条后放行）
```
