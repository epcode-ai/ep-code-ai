# 配置漂移审计

## 用途

dev / staging / prod 三个环境的配置本应保持结构一致,但实际总会漂移。AI 批量对比找异常。

## 使用场景

- 季度/月度配置审计
- 故障后怀疑是配置问题
- 新环境搭建后对齐

## Prompt 本体

```markdown
我有多个环境的配置文件,帮我找出异常。

## Dev 环境
```yaml
[粘贴 dev config]
```

## Staging 环境
```yaml
[粘贴 staging config]
```

## Prod 环境
```yaml
[粘贴 prod config]
```

## （可选）Canary/灰度环境
```yaml
[粘贴]
```

## 任务

### 1. 列出所有差异
对每个配置项,如果环境间值不同,列出:
| 配置项 | Dev | Staging | Prod | 评价 |

### 2. 差异分类

对每个差异判断:
- ✅ **合理差异**: 应该不同（如 URL、实例数、密钥）
- ⚠️ **可疑差异**: 不应该这么不一致（如超时、重试次数、阈值）
- ❌ **明显错误**: 环境之间缺了某项（可能漏配）

### 3. 异常清单（重点）

对每个可疑/错误项:
- 位置: [配置路径]
- 具体差异: [描述]
- 为什么是异常: [理由]
- 建议修复: [方案]

### 4. 配置一致性评分

给出一个分数（0-100 分）:
- 结构一致性: X
- 命名一致性: X
- 关键参数一致性: X
- 总分: X

### 5. 最佳实践违反

检查:
- [ ] 敏感信息是否硬编码（密钥、密码）
- [ ] 是否有硬编码的 IP/host
- [ ] 超时/重试配置是否合理
- [ ] 是否有未使用的配置项
- [ ] 是否缺少默认值
- [ ] 是否有过于宽松的权限

### 6. 配置治理建议

- 哪些配置应该**从代码里抽出**到配置中心
- 哪些配置应该**统一**（所有环境共用）
- 哪些配置应该**加校验**（启动时 fail fast）
- 哪些环境**缺少某项**而应该加上

## 输出格式

结构化的 Markdown 报告，以便贴到 Issue 跟踪。
```

## 使用示例

**输入**: 3 个环境的 application.yml（各 200 行）

**输出**:
- 发现 12 处差异:合理 8 + 可疑 3 + 错误 1
- 错误: prod 环境缺少 `redis.cluster` 配置
- 可疑: staging 超时 5s、prod 30s（应该对齐）
- 建议: 把超时配置统一提到中心化管理
