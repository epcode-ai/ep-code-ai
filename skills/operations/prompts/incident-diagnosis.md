# 故障诊断对话

## 用途

故障处置时，IC 可以开一个 Claude 对话窗口作为"副驾驶"，边查边问。

## 使用场景

- SEV-1/SEV-2 故障实时处置
- 复杂故障的诊断辅助
- 值班新人 backup

## Prompt 本体（首轮）

```markdown
我正在处理一个生产故障，需要你辅助诊断。我们将进行多轮对话。

## 故障现象
[描述症状：告警内容、用户反馈、监控异常]

## 已知信息
- 服务: [服务名]
- 服务拓扑: [简述]
- 最近发布: [时间 + 版本 + 改动]
- 最近配置变更: [列表]
- 最近依赖变化: [列表]
- 当前严重度: SEV-[X]

## 已尝试过
1. [动作 1 + 结果]
2. [动作 2 + 结果]

## 当前线索
```
[粘贴关键日志 / 监控图 / 告警]
```

## 请你
1. 列出接下来应该检查的 3-5 个方向，**按价值排序**
2. 每个方向给出**具体的检查命令或查询**
3. 对每个可能的结果,提前告诉我应该做什么

## 约束
- 优先给出**能在 5 分钟内执行**的检查
- 不要给出破坏性操作（重启、回滚）除非明确要求
- 如果需要我提供更多信息,明确说要什么

我会根据你的建议去查，边查边把结果贴给你，我们迭代诊断。
```

## 续聊模板

```markdown
## 上一步结果
[粘贴你查到的结果]

## 新的观察
[如果有新线索]

## 继续指导我下一步
```

## 使用示例

**场景**: SEV-1 订单服务异常

**对话 1**（首轮）:
- 用户: [故障现象 + 已知信息]
- AI: "按价值建议查: 1) 最近 30 分钟发布 2) DB 连接池 3) 下游依赖健康"

**对话 2**:
- 用户: "查了,30 分钟前发布了 v1.2.1，改动是连接池配置"
- AI: "这个改动很可疑。下一步: `kubectl logs ... | grep 'pool'`，看有没有连接池相关错误"

**对话 3**: ...

**最终**: 定位并回滚

## 使用注意

### DO

- ✅ 保留对话记录,便于事后复盘
- ✅ AI 的建议**要验证再执行**
- ✅ 不确定时告诉 AI "不确定，给我更多可能"

### DON'T

- ❌ 不要让 AI 直接执行命令（它只能建议）
- ❌ 不要粘贴涉密信息（客户数据、内部 IP）
- ❌ 不要跳过 AI 的"请提供更多信息"就强行要答案
