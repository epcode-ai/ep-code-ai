# Runbook 生成

## 用途

把散落的故障处置经验、告警规则，结构化成可执行的 Runbook。

## 使用场景

- 故障复盘后沉淀 Runbook
- 新告警规则上线同步建立 Runbook
- 老值班知识文档化

## Prompt 本体

```markdown
基于以下信息,帮我生成一份 Runbook。

## 场景信息

### 告警/现象
- 告警名:
- 告警规则: [Prometheus/Grafana 表达式]
- 典型现象: [症状描述]
- 用户感知:

### 历史故障（可选）
[粘贴过往同类故障的复盘摘要]

### 已知处置经验（可选）
[从老员工那里收集的经验]

### 服务上下文
- 服务名:
- 部署方式: [K8s/VM/Docker]
- 关键依赖: [列表]

## Runbook 结构要求

参照 `templates/operations/runbook-template.md`:

1. **故障现象**: 告警、用户表现、监控异常
2. **严重度评估**: 什么情况是 SEV-1/2/3
3. **快速定位**（5 分钟内 3 步）
4. **处置方案**（至少 3 个,按适用场景区分）
5. **常见根因**（按频率排序）
6. **升级路径**
7. **事后必做**

## 输出要求

- **具体可执行**: 给出真实可跑的命令（不是"查一下日志"这种空话）
- **分条件**: 每个处置方案说明适用什么情况
- **有回滚预案**: 每个处置动作后"如果没改善怎么办"
- **有成本评估**: 每个动作的影响面
- **有验证方法**: 怎么确认问题真的解决

## 注意

- AI 生成的 Runbook **必须经过值班人员评审和演练**才能正式使用
- 建议标注"v1.0 Draft - AI 生成，待人工验证"
- 每次故障后根据实际经验更新
```

## 使用示例

**输入**: "HighMemoryUsage" 告警 + 2 次历史故障经验

**输出**: 完整 Runbook 草稿，包括：
- 如何区分真 OOM vs 假告警
- 扩容/重启/降级三种处置
- 查看 heap dump 的具体命令
- 升级到 SEV-1 的条件
