# 混沌演练设计

## 用途

系统越复杂越需要混沌工程。AI 帮助设计覆盖面更全的演练方案。

## 使用场景

- 新系统上线前的韧性验证
- 定期演练规划
- 架构重大变更后

## Prompt 本体

```markdown
帮我设计一组混沌工程实验。

## 系统拓扑
```
[描述服务间依赖，可用 ASCII 图或 Mermaid]
```

## 关键服务列表
| 服务 | 职责 | 依赖 | 是否核心 |
|------|------|------|---------|

## 演练目标

选择要验证的场景（可多选）:
- [ ] 单实例挂掉（扩容/重启）
- [ ] 某 AZ/机房故障
- [ ] 某依赖完全不可用
- [ ] 某依赖慢（慢响应）
- [ ] 网络丢包/延迟
- [ ] DNS 故障
- [ ] CPU 满载
- [ ] 内存满载
- [ ] 磁盘满
- [ ] 证书过期
- [ ] 时钟漂移
- [ ] 配置错误

## 约束

- 演练时间窗口: [工作时间 / 夜间]
- 可接受的"爆炸半径": [如 < 1% 用户]
- 回滚时间要求: [< N 分钟]
- 预演要求: 先预发布演练

## 输出

对每个场景给出：

### 场景 N: [名称]

#### 实验设计

- **注入方式**: [用 Chaos Mesh / Gremlin / 自研工具的语法]

```yaml
# 示例：Chaos Mesh YAML
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
...
```

#### 预期行为

正常情况下,系统应该:
1. [行为 1]
2. [行为 2]

#### 验证方法

- 监控指标: [看哪些大盘 + 预期曲线]
- 业务指标: [核心转化率等]
- 日志: [关注什么日志]
- 用户感知: [应该/不应该感知到什么]

#### 爆炸半径控制

- 影响范围: [限定在 X%]
- 自动停止条件: [触发什么阈值就自动停]
- 手动停止: [谁可以停 + 怎么停]

#### 演练前准备

- [ ] 通知相关方（如非生产）
- [ ] 确认回滚机制可用
- [ ] 应急联系人到岗
- [ ] 监控大盘打开

#### 演练时长

- 预热: [X 分钟]
- 注入: [X 分钟]
- 观察: [X 分钟]
- 恢复: [X 分钟]

#### 预期发现

- 可能暴露的问题: [列表]
- 如果没暴露出问题: [也是一种结果,说明此处韧性足够]

## 演练节奏建议

1. 先做哪些（从小爆炸半径开始）
2. 递进到哪些
3. 什么情况下停止继续

## 演练后的事

- 演练报告模板
- 发现的问题如何跟踪
- 如何更新 Runbook
```

## 使用示例

**输入**: 微服务系统拓扑 + 6 个演练目标

**输出**: 6 个详细实验方案（YAML + 验证清单 + 爆炸半径控制）
