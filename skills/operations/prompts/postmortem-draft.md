# 故障复盘报告初稿

## 用途

故障结束后写复盘报告往往耗时 2-4 小时。让 AI 生成初稿，人工补充判断部分。

## 使用场景

- 故障后 24-48h 内
- 月度质量报告的素材准备

## Prompt 本体

```markdown
基于以下信息，生成故障复盘报告初稿。

## 重要原则
遵循 **Blameless Culture**（对事不对人）:
- 不指名道姓地追责
- 把"个人失误"转化为"流程漏洞"
- 找**系统性问题**,而不是找替罪羊

## 故障基本信息

- 故障 ID: INC-YYYY-NNNN
- 严重度: SEV-[X]
- 发生时间: YYYY-MM-DD HH:MM
- 恢复时间: YYYY-MM-DD HH:MM
- 影响范围: [用户数 / 业务]

## 原始材料

### 时间线（故障群关键消息）
```
[粘贴故障群里的关键消息 + 时间戳，按时间顺序]
```

### 监控数据
[描述或贴关键指标的变化]
- 错误率: 基线 0.1%，峰值 2.5%
- P99 延迟: 基线 180ms，峰值 1200ms
- QPS: ...

### 处置动作
[列出执行过的操作]
1. HH:MM 执行了 X
2. HH:MM 执行了 Y

### 已知原因
[如果已经初步定位]

## 输出结构

参照 `templates/operations/postmortem-template.md`:

1. **摘要**（TL;DR）
2. **影响评估**（量化）
3. **时间线**（精确到分钟）
4. **故障经过**（叙述体）
5. **根因分析**（5 Why）
6. **做得好的地方**
7. **做得不好的地方**
8. **改进项**（SMART 原则）
9. **教训**

## 输出要求

- 对事不对人（不要出现"张三没注意"）
- 5 Why 要深挖到**流程/文化**层，不能只停在"代码 Bug"
- 改进项必须**具体可执行**（不要"加强监控"这种虚词）
- 对每个改进项,预留"负责人 / 截止日期 / 状态"字段
- 末尾标注"**本报告由 AI 生成初稿,需参与人员逐条核对**"

## 不确定的地方
用 ⚠️ 标注，并给出"需要 IC 确认"的提示。
```

## 使用示例

**输入**: 故障群 200 条消息 + 监控截图描述

**输出**: 10 页复盘报告初稿,结构完整,包括 5 Why + 改进项清单
