# 开发模块在企业项目生命周期中的定位与体系化建设

开发是"把业务想法变成真实代码"的核心环节。代码质量决定产品质量,开发流程决定团队节奏。本篇从三个层面展开：输入输出定义、Claude 工具化赋能、与多角色的协作规范。

---

## 一、开发模块的输入输出定义

开发在每个阶段都有明确的"契约"：

### 各阶段输入输出矩阵

| 阶段 | 输入（上游交付物） | 输出（开发交付物） | 上游角色 | 下游角色 |
|------|------------------|------------------|---------|---------|
| 需求接收 | PRD、用户故事、验收标准 | 需求澄清清单、工期评估 | 产品 | 开发内部 |
| 技术方案 | 需求包、架构约束、非功能要求 | 概要设计、ADR、API 契约 | 产品/架构 | 测试/运维 |
| 详细设计 | 概要设计 | 模块详细设计、DB schema | 架构 | 开发同事 |
| 编码实现 | 设计文档、API 契约 | 可编译代码、单元测试、接口 Mock | 开发 | 开发同事 / 测试 |
| 代码评审 | 本地开发完成的 PR | 评审意见、修订提交 | 同事评审者 | 开发 |
| 自测 + CI | 合并到测试分支的代码 | 自测报告、CI 绿灯 | 开发 | 测试 |
| 提测 | 合格的构建 + 提测申请单 | 测试接受,进入测试阶段 | 开发 | 测试 |
| 修复回归 | Bug 报告 | 修复 PR + 说明 | 测试 | 测试（回归） |
| 发布支持 | 准出的版本 | 发布上线 + 观察 | 测试/运维 | 运维 |

### 关键原则

**"接口先于代码，测试先于实现"** — 开发侧的两条底线。具体来说：

- **准入标准（Entry Criteria）**：PRD 通过可测性评审;技术方案有 ADR 支撑;工期已评估
- **准出标准（Exit Criteria）**：CI 绿灯;单元测试覆盖率 ≥ 80%;接口文档与代码一致;自测完整

## 二、用 Claude 构建开发工具链

Claude 可以在开发的每个环节提供工具化支持：

### 1. 方案设计阶段 — 多方案对比

```
帮我在以下候选方案中做选择。

## 业务场景
[描述业务背景 + 关键约束]

## 我们的现状
- 团队规模 / 技术栈 / 运维能力 / 时间窗口

## 候选方案
- 方案 A: [简述]
- 方案 B: [简述]
- 方案 C: [简述]

## 对比维度（按优先级）
性能 / 实现复杂度 / 维护成本 / 可扩展性 / 学习曲线 / 社区活跃度 / 风险

## 输出
1. 对比矩阵（每维度打分 1-5）
2. 每方案决定性优缺点
3. 推荐方案 + ADR 格式的决策记录
```

### 2. 代码评审阶段 — 自动审查

```
你是一位资深代码评审员。评审以下 diff。

## 评审维度
1. 正确性: 逻辑 / 边界 / 异常 / 并发
2. 安全: 注入 / XSS / 敏感信息 / 越权
3. 性能: N+1 / 循环 IO / 缓存 / 资源泄漏
4. 可读性: 命名 / 函数长度 / 嵌套
5. 测试: 覆盖 / 命名 / Mock

## 输出
对每个问题输出:
| 文件:行号 | 严重度 | 问题 | 修复建议 |

严重度:
- 🔴 must-fix
- 🟡 should-fix
- 🟢 nit
```

### 3. 接口文档生成 — 从代码反向

```
根据以下代码生成 Markdown 格式的接口文档。

## 文档结构
1. 基本信息（方法/路径/认证）
2. 请求参数（Headers / Path / Query / Body）
3. 响应（成功 + 业务错误码）
4. 示例
5. 测试用例提示（边界/异常/安全）

## 代码
[粘贴路由 + 处理函数 + schema]

## 要求
- 类型信息从代码提取
- 错误码从异常处理代码提取
- 必填/可选明确标注
- 末尾标注 "AI 生成,需人工核对"
```

### 4. 单元测试生成 — 从接口契约

```
根据以下接口契约,生成 [pytest / jest / junit] 单元测试。

## 覆盖要求
1. 正常流 / 异常流 / 边界值 / 数据类型异常
2. 并发场景 / 幂等性 / 安全场景（如涉及）

## 代码风格
- AAA 模式（Arrange / Act / Assert）
- 命名: `should_<行为>_when_<条件>`
- 一个测试测一件事
- Mock 外部依赖

## 接口契约
[粘贴 API 文档]

## 输出
- 完整可执行测试代码
- 每个测试一句注释
- 需要 mock 的外部依赖清单
```

### 5. SQL 优化 — 慢查询分析

```
你是一位资深 DBA。分析这个慢查询,给优化方案。

## SQL
[粘贴]

## EXPLAIN 输出
[粘贴]

## 表信息
大小、索引、主键

## 输出
1. 慢的根本原因
2. 优化方案（按改动成本从小到大）
3. 推荐方案及理由
4. 验证方法
5. 需要注意的副作用
```

### 6. 代码重构 — 保持行为不变

```
帮我重构以下代码。

## 重构目标
降低复杂度 / 提升可读性 / 提取复用 / 便于测试

## 约束
1. 不改公共接口
2. 不引入新依赖
3. 保持原有测试通过

## 输出
1. 重构后代码
2. 改动清单（before → after + 理由）
3. 风险点（需人工 review）
```

### 7. 错误排查 — Stack Trace 定位

```
帮我排查这个错误。

## 错误信息
[完整 stack trace]

## 复现步骤
## 相关代码

## 我已尝试过

## 输出
1. 最可能的原因（top 3,按概率）
2. 每个原因的验证方法
3. 修复方案
4. 长期改进建议
```

### 8. 代码翻译 — 跨语言迁移

```
把这段 [源语言] 代码翻译成 [目标语言]。

## 要求
1. 功能完全一致
2. 使用目标语言惯用法
3. 补充类型声明
4. 异常处理按目标语言习惯

## 输出
```

## 三、与多角色的协作规范 — 生命周期串联

### 核心思路：用"契约"串联角色，用"门禁"保证质量

```
┌─────────┐   PRD + AC      ┌─────────┐
│ 产品/业务 │ ─────────────► │         │
└─────────┘                 │         │  设计 + API 契约
                            │  开发   │ ──────────►  测试
┌─────────┐   技术方案      │         │
│  架构/PM │ ─────────────► │         │  可发布版本
└─────────┘                 │         │ ──────────►  运维
         ◄── Bug 报告 ──── │         │
         ◄── 线上反馈 ──── │         │
                            └─────────┘
```

### 具体规范建议

**1. 与产品的契约**

| 项目 | 规范 |
|------|------|
| PRD 接收 | PRD 必须通过可测性评审才接受 |
| 需求澄清 | 所有疑问书面化,24h 内回应 |
| 变更处理 | 只接受走 CR 流程的变更,口头变更不算数 |

**2. 与测试的契约**

| 项目 | 规范 |
|------|------|
| 提测标准 | 见"提测达标"（不合格直接退回） |
| 接口文档 | 与代码一致,代码改动必须同步文档 |
| Bug 响应 | P0 4h 内;P1 当天;P2 2 工作日 |
| 环境保障 | 开发负责测试环境可用 |

**3. 与运维的契约**

| 项目 | 规范 |
|------|------|
| 部署文档 | 发版前提交,包含配置/依赖/回滚 |
| 监控埋点 | 发版前约定好,与运维对齐 |
| 灰度支持 | 随时待命,协助排查灰度问题 |
| 技术债 | 定期（季度）向运维同步架构演进 |

**4. 与架构的契约**

| 项目 | 规范 |
|------|------|
| ADR | 核心技术决策必须有 ADR |
| 技术方案评审 | 架构委员会批准才能进入详细设计 |
| 跨系统影响 | 动全局组件前必须评审 |

### 用 Claude 落地这套规范的实际做法

**建立模板库**：
- 概要设计文档模板
- ADR 模板
- 代码评审 Checklist
- Release Note / CHANGELOG 模板
- 依赖升级提案模板
- API 契约 Markdown 模板

**建立 Skill**：
- `/code-review` — 代码评审辅助
- `/api-doc-from-code` — 反向生成接口文档
- `/unit-test-from-contract` — 根据契约生成单测
- `/sql-optimize` — SQL 优化建议
- `/refactor` — 代码重构
- `/error-triage` — 错误排查
- `/tech-decision` — 技术方案对比（输出 ADR）
- `/translate-code` — 代码翻译
- `/log-analysis` — 日志分析

## 四、总结：开发的核心定位

开发模块的本质是**把确定的需求转化为稳定的代码**：

1. **向上游要求确定性**：没有可测需求不接受
2. **向下游交付稳定性**：接口文档先行,单测保底,CI 守关
3. **面对变化保持韧性**：架构可扩展,代码可维护,决策可追溯（ADR）
4. **持续消化技术债**：不追求完美,但不积累腐烂

最终目标：**开发不是"功能输出机",而是稳定可信的交付伙伴**。

## 五、开发生命周期的规范化运行框架

### 5.1 开发的"七阶段门禁模型"

```
 Gate In              阶段              Gate Out            下一阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 PRD已批准      ──►  ① 需求接收    ──►  工期评估 + 澄清清单   ──►

 需求清晰      ──►  ② 技术方案    ──►  概要设计 + ADR       ──►

 方案批准      ──►  ③ 详细设计    ──►  详细设计 + API 契约   ──►

 契约确定      ──►  ④ 编码实现    ──►  可编译 + 单测通过     ──►

 代码就绪      ──►  ⑤ 代码评审    ──►  至少 1 人 approve     ──►

 评审通过      ──►  ⑥ 自测 + CI   ──►  CI 绿灯 + 自测报告    ──►

 自测达标      ──►  ⑦ 提测        ──►  测试接受,进入下阶段   ──►  测试
```

### 5.2 每个阶段的具体规范

**阶段①：需求接收**

```yaml
gate_in:
  必须项:
    - PRD v1.0（已通过跨部门评审）
    - 可测性评审通过

deliverables:
  - 需求澄清清单
  - 工期评估（含风险）

gate_out:
  - 所有疑问已回答
  - 可分到具体开发
```

**阶段②：技术方案**

```yaml
gate_in:
  必须项:
    - 清晰需求 + 非功能要求

deliverables:
  - 概要设计文档（方案选型 + 架构图 + 时序图）
  - ADR（关键决策记录）
  - API 契约（Markdown 格式）

gate_out:
  - 架构委员会批准（大改动）
  - 测试 + 运维评审通过
```

**阶段③：详细设计**

```yaml
deliverables:
  - 模块详细设计
  - 数据库 schema + migration
  - 关键流程时序图

gate_out:
  - 开发组内评审通过
```

**阶段④：编码实现**

```yaml
编码规范:
  - 命名、注释、风格符合约定
  - 每个接口有单元测试
  - 单测覆盖率 ≥ 80%

工具:
  - 本地 lint + format
  - pre-commit hook
```

**阶段⑤：代码评审（关键门禁）**

```yaml
gate_in:
  必须项:
    - PR 描述完整
    - CI 绿灯
    - 自测截图

评审 Checklist:
  - 正确性 / 安全 / 性能 / 可读性 / 测试 / 文档

gate_out:
  - 至少 1 人 approve
  - 所有 must-fix 问题已解决
```

**阶段⑥：自测 + CI**

```yaml
gate_in:
  必须项:
    - 代码已合到 release 分支
    - CI 全绿

自测清单:
  - 冒烟测试通过
  - 本次变更点全部自测
  - 边界/异常场景自测

deliverables:
  - 自测报告
  - 测试环境已部署
```

**阶段⑦：提测**

```yaml
gate_in（提测达标）:
  - CI 绿灯 + 单测覆盖率
  - 冒烟通过
  - 提测申请单完整
  - 接口文档与代码一致
  - 测试环境可访问
  - 数据库/配置变更脚本齐全

deliverables:
  - 提测申请单
  - 部署文档
```

## 七、跨角色协作的"契约模板"

### 7.1 与产品的契约

```yaml
# PRD 接收标准

必须通过:
  - 可测性评审
  - 业务规则完整
  - 非功能要求量化

工期评估承诺:
  - 评估失误 ± 20% 以内
  - 大块时间变化 48h 内通知
```

### 7.2 与测试的契约

详见全局[提测达标](../04-testing/04-gates/submission-gate.md)。核心硬要求：

- ✅ 代码已合并到 release 分支
- ✅ CI 流水线绿灯
- ✅ 单元测试覆盖率 ≥ 80%
- ✅ 冒烟测试通过
- ✅ 提测申请单完整
- ✅ 接口文档与代码一致
- ✅ 测试环境已部署且可访问

### 7.3 与运维的契约

```yaml
# 部署契约

发版前必须提供:
  - 部署文档
  - 回滚方案
  - 监控埋点约定
  - 配置变更清单
  - 数据库 migration + rollback 脚本

发版时:
  - 开发 standby
  - P0 问题 5 分钟响应
```

## 八、开发知识库构建

用 Git + Markdown + AI 构建开发团队的组织记忆：

```
实体类型:

[Project]  ──has──► [Module]
[Module]   ──owned_by──► [Developer]
[Module]   ──has_adr──► [ADR]
[Module]   ──implements──► [API]
[API]      ──tested_by──► [TestCase]
[Bug]      ──affects──► [Module]
[Bug]      ──fixed_by──► [Commit]
```

**好处**：
- **架构可追溯**：为什么这样设计? 查 ADR
- **Bug 知识沉淀**：同类 Bug 不重复犯
- **代码传承**：新人快速理解模块

## 十、总结：开发负责人的行动清单

| 优先级 | 行动项 | 预期收益 |

| **P2** | 依赖管理流程（Dependabot） | 安全漏洞及时补 |
| **P2** | 自动 Release（semantic-release） | 发版工作量下降 |

## 十二、名词解释

| 术语 | 定义 |
|------|------|
| **ADR** | Architecture Decision Record,架构决策记录。轻量级的单次决策存档文档 |
| **PR / MR** | Pull Request（GitHub）/ Merge Request（GitLab）,代码合并请求 |
| **CI** | Continuous Integration,持续集成。每次代码变更自动跑构建、测试 |
| **CD** | Continuous Delivery/Deployment,持续交付/部署 |
| **API 契约** | 接口规范。本项目推荐用 Markdown 格式替代 Swagger |
| **Conventional Commits** | 约定式提交,规定 `<type>(<scope>): <subject>` 格式 |
| **SemVer** | Semantic Versioning,语义化版本。格式 `v<MAJOR>.<MINOR>.<PATCH>` |
| **SBOM** | Software Bill of Materials,软件物料清单,列出所有依赖 |
| **单元测试** | 对最小可测单元（函数/方法）的测试 |
| **代码覆盖率** | 被测试覆盖的代码比例（行/分支/函数） |
| **AAA 模式** | Arrange-Act-Assert,单元测试的标准结构 |
| **N+1 查询** | ORM 常见陷阱：一次查询引发 N 次子查询 |
| **N+1 问题** | 同上,性能反模式 |
| **幂等性** | 多次执行同一操作结果相同（PUT/DELETE 天然幂等） |
| **Mock** | 测试替身,模拟外部依赖的行为 |
| **Lint** | 静态代码检查工具 |
| **pre-commit hook** | Git 钩子,提交前自动执行检查 |
| **Squash merge** | 合并 PR 时压成一个 commit |
| **Rebase** | 重新基准合并,保持线性历史 |
| **Hotfix** | 线上紧急修复分支,从 main 拉出 |
| **Breaking Change** | 破坏性变更,不向下兼容 |
| **Dependabot** | GitHub 官方依赖自动升级机器人 |
| **PSScriptAnalyzer** | PowerShell 静态分析工具 |
| **shellcheck** | Shell 脚本静态检查工具 |
| **CVE** | Common Vulnerabilities and Exposures,公开安全漏洞编号 |
