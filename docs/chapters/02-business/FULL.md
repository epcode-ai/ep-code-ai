# 业务模块在企业项目生命周期中的定位与体系化建设

业务模块是整条研发流水线的**源头**。源头干净,下游才有的做;源头混浊,再好的技术团队也是在补窟窿。本篇从三个层面展开：输入输出定义、Claude 工具化赋能、与多角色的协作规范。

---

## 一、业务模块的输入输出定义

业务在每个阶段都有明确的"契约"：

### 各阶段输入输出矩阵

| 阶段 | 输入（上游交付物） | 输出（业务交付物） | 上游角色 | 下游角色 |
|------|------------------|------------------|---------|---------|
| 市场/客户洞察 | 战略目标、客户反馈、竞品动态 | 机会清单、优先级评估 | 业务/市场 | 产品 |
| 需求收集 | 需求池、访谈、用户调研 | 需求说明、用户画像、场景 | 客户/运营 | 产品内部 |
| 需求分析 | 需求说明、业务规则 | 用户故事、业务规则表、决策表 | 产品 | 产品内部 |
| PRD 编写 | 用户故事、验收标准、原型 | PRD v1.0、数据字典、流程图 | 产品 | 开发/测试 |
| 需求评审 | PRD、用户故事 | 评审纪要、可测性反馈、变更清单 | 产品 | 开发/测试/项目经理 |
| 变更管理 | 客户新诉求、开发反馈 | CR（变更请求单）、影响评估、决策记录 | 产品 | 全员 |
| 上线验收 | 测试准出报告、发布的产品 | 业务验收报告、用户反馈汇总 | 产品 | 运营/业务 |

### 关键原则

**"没有可测的需求，就没有可交付的产品"** — 业务守住产品价值的第一道防线。具体来说：

- **准入标准（Entry Criteria）**：客户诉求必须可复述;业务规则必须可枚举
- **准出标准（Exit Criteria）**：每条需求点有验收标准;异常场景 ≥ 80% 覆盖;数据规则明确;性能指标量化

## 二、用 Claude 构建业务分析工具链

Claude 可以在业务侧的每个环节提供工具化支持：

### 1. 需求调研阶段 — 访谈记录结构化

```
你是一位资深产品经理。我有一份用户访谈的原始记录，帮我整理成结构化的需求文档。

## 输入
[访谈录音转写]

## 输出结构
1. 访谈基本信息
2. 核心痛点（按重要度排序，附原话引用）
3. 未被满足的需求（As a... I want... So that...）
4. 现有方案的评价
5. 潜在机会点
6. 遗留问题
7. 金句摘录

## 约束
- 区分【事实】vs【推测】
- 原话引用要带引号和出处
- 不编造没说过的内容
```

**产出**：2 页结构化需求文档,可以直接作为 PRD 的背景章节。

### 2. 需求分析阶段 — 用户故事拆解

```
你是一位资深敏捷教练。帮我把这个大故事拆成 3-7 个可独立交付的子故事。

## 要求
1. 每个子故事满足 INVEST 原则
2. 每个子故事输出:
   - 标题
   - As a / I want / So that
   - 至少 3 条验收标准（Given-When-Then）
   - Story Point 估算
3. 标注子故事之间的依赖关系

## 大故事
[粘贴原始故事]
```

### 3. PRD 编写阶段 — 可测性自检

```
你是一位资深测试架构师。从"可测性"角度扫描这份 PRD，找出：

1. 哪些需求点缺少验收标准（无 Given-When-Then）
2. 哪些描述过于模糊（如"用户体验好"、"响应快"）
3. 哪些边界条件未明确
4. 哪些异常场景缺失
5. 哪些数据规则不清
6. 哪些性能指标未量化

## 输出
表格: 问题编号 | 章节 | 问题类型 | 具体问题 | 建议补充

## PRD
[粘贴]
```

### 4. 业务规则阶段 — 自然语言转决策表

```
把下面的业务描述转成决策表。

## 要求
1. 列出所有条件变量
2. 枚举所有可能的组合
3. 输出 Markdown 表格
4. 标注未明确的情况（需要产品确认）
5. 标注相互冲突的规则

## 业务描述
[粘贴]
```

### 5. 变更管理阶段 — 影响评估

```
你是一位资深系统分析师。评估这个变更的影响。

## 分析维度
1. 功能影响
2. 数据影响
3. 接口影响
4. 性能影响
5. 安全影响
6. 用户体验影响
7. 测试影响
8. 下游影响（依赖此功能的其他团队/系统）
9. 时间影响

## 变更描述
[粘贴 CR]

## 输出
| 维度 | 影响 | 严重度 | 具体说明 |
总结: 总体评级 + 建议接受/有条件/驳回
```

### 6. 竞品分析阶段 — 结构化对比

```
帮我做竞品功能对比分析。

## 目标功能
[功能简述]

## 候选竞品
[竞品清单 + 已掌握的资料]

## 对比维度
是否有此功能 / 实现方式 / 交互特点 / 定价 / 优势 / 劣势

## 输出
1. 对比矩阵
2. 差异化机会点
3. 必须补齐的功能
4. 建议放弃的方向
```

### 7. 需求冲突检测 — 长文档扫描

```
你是一位资深产品分析师。扫描以下 PRD，找出前后矛盾或逻辑冲突。

## 检查维度
1. 术语一致性（同一概念不同章节）
2. 规则一致性（允许与禁止并存）
3. 数值一致性（24h vs 12h）
4. 流程闭合性（有进无出）
5. 权限一致性

## PRD
[粘贴]

## 输出
| # | 严重度 | 涉及章节 | 冲突描述 | 建议协调 |
```

## 三、与多角色的协作规范 — 生命周期串联

### 核心思路：用"契约"串联角色，用"门禁"保证价值

```
┌──────────┐   市场信号 / 客户诉求   ┌─────────┐
│ 业务/市场  │ ──────────────────► │         │
└──────────┘                      │         │
                                  │         │  PRD + 验收标准
┌─────────┐   客户访谈 / 运营反馈    │  业务/PM │ ──────────►  开发 / 测试
│  运营/客服 │ ──────────────────► │         │
└─────────┘                      │         │
         ◄──── 可测性反馈 ──────── │         │  CR / 变更决策
         PRD 评审意见 ──────────► │         │ ──────────►
                                  └─────────┘
```

### 具体规范建议

**1. 与业务/市场方的契约**

| 项目 | 规范 |
|------|------|
| 输入格式 | 需求池必须有 ID、来源、优先级、提出日期 |
| 评审机制 | 季度优先级会,月度需求池清理 |
| 变更管理 | 战略方向变化 24 小时内通知 PM |

**2. 与开发的契约**

| 项目 | 规范 |
|------|------|
| PRD 交付 | 开发介入前 2 个工作日提供 PRD v1.0 |
| 验收标准 | 必须 Given-When-Then 格式,覆盖核心流 + 异常 |
| 变更沟通 | 开发阶段变更走正式 CR,影响评估齐全才决策 |
| 业务规则 | 复杂逻辑用决策表,不要大段文字 |

**3. 与测试的契约**

| 项目 | 规范 |
|------|------|
| 可测性 | PRD 必须通过可测性评审（见测试篇） |
| 场景覆盖 | 异常场景必须列举 |
| 数据规则 | 字段类型、长度、格式、必填必须明确 |

**4. 与运维/运营的契约**

| 项目 | 规范 |
|------|------|
| 上线验收 | 产品签署业务验收后才算"完成" |
| 运行反馈 | 运营提供月度用户行为数据,反哺需求 |
| 线上问题 | 产品参与客诉处理,判断是否是需求问题 |

### 用 Claude 落地这套规范的实际做法

**建立模板库**：
- PRD 模板
- 用户故事模板
- 业务规则模板（决策表 + 状态机）
- 变更请求（CR）模板
- 竞品分析模板
- 访谈记录模板

**建立 Skill**：
- `/prd-check` — PRD 可测性扫描
- `/user-story-split` — 大故事自动拆分
- `/business-rule-to-table` — 自然语言转决策表
- `/requirement-conflict` — 需求冲突检测
- `/change-impact` — 变更影响评估
- `/interview-summary` — 访谈记录整理
- `/competitor-analysis` — 竞品对比

## 四、总结：业务的核心定位

业务模块的本质是**把混沌的市场/客户信号转化为确定的、可执行的需求**：

1. **向上游承接价值**：把商业目标、客户诉求翻译成"产品语言"
2. **向下游交付确定性**：让开发/测试不需要猜,按 PRD 就能干活
3. **持续管理变化**：需求会变,但变的过程必须可控
4. **数据驱动决策**：不凭感觉,用用户数据和反馈闭环

最终目标：**业务不是"传声筒",而是价值的翻译官和把关人**。

## 五、业务生命周期的规范化运行框架

### 5.1 业务的"六阶段门禁模型"

```
 Gate In          阶段             Gate Out           下一阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 有信号来   ──►  ① 调研洞察  ──►  需求卡片 + 优先级    ──►

 进入 TODO ──►  ② 需求分析  ──►  用户故事 + 业务规则  ──►

 可以写 PRD ──► ③ PRD 编写  ──►  PRD v1.0            ──►

 v1.0 就绪 ──►  ④ 跨部门评审 ──►  可测性通过 + 已批准  ──►

 批准 / 冻结 ──► ⑤ 变更管理  ──►  CR 流程 / 版本控制   ──►

 上线后    ──►  ⑥ 业务验收  ──►  验收报告 + 迭代计划   ──►  Done
```

### 5.2 每个阶段的具体规范

**阶段①：调研洞察**

```yaml
stage: 调研洞察
trigger: 有业务信号（客户反馈 / 市场机会 / 战略任务）

gate_in:
  必须项:
    - 信号来源明确（谁、什么渠道、什么时候）
    - 初步假设（解决什么问题）

deliverables:
  - 需求卡片（轻量）:
      内容: 标题 / 问题 / 受众 / 价值 / 规模估计
      用途: 进需求池,等排期

gate_out:
  - 需求卡片已进池
  - 有人认领
```

**阶段②：需求分析**

```yaml
stage: 需求分析
trigger: 需求卡片被排进当期迭代

gate_in:
  必须项:
    - 需求卡片完整
    - 产品 Owner 确认接手

deliverables:
  - 用户故事集（INVEST 合规）
  - 业务规则表（决策表形式）
  - 流程图 / 状态机（如适用）

gate_out:
  - 故事已拆到可估算粒度（≤ 8 Story Point）
  - 业务规则无冲突
```

**阶段③：PRD 编写**

```yaml
stage: PRD 编写
trigger: 需求分析完成

deliverables:
  - PRD v0.x（草稿）
  - 数据字典
  - 交互流程图（Mermaid 或 Figma）

gate_out:
  - PRD 通过产品内部 review
  - 可测性自检通过（用 skill `/prd-check`）
```

**阶段④：跨部门评审**

```yaml
stage: 跨部门评审
trigger: PRD v0.x 就绪

gate_in:
  必须项:
    - PRD v0.x
    - 评审邀请发出（提前 2 工作日）
    - 测试的可测性评审意见已收集（会前）

deliverables:
  - 评审纪要
  - 改进清单 + Owner

gate_out:
  - 所有阻塞问题已解决
  - PRD 升到 v1.0,冻结
```

**阶段⑤：变更管理**

```yaml
stage: 变更管理
trigger: 冻结后出现变化诉求

gate_in:
  必须项:
    - 变更请求单（CR）
    - 影响评估（skill `/change-impact`）

decision_path:
  - 批准立即执行
  - 批准延后到下版本
  - 驳回

gate_out:
  - 决策已记录
  - 相关方已通知
```

**阶段⑥：业务验收**

```yaml
stage: 业务验收
trigger: 测试准出 + 运维上线

deliverables:
  - 业务验收报告（验收标准逐条过）
  - 用户反馈初步收集

gate_out:
  - 所有验收标准通过
  - 下一轮迭代计划形成
```

## 七、跨角色协作的"契约模板"

### 7.1 与开发的契约

```yaml
# PRD 交付契约

必须字段:
  - 背景 / 目标 / 非目标
  - 用户故事（含 AC）
  - 业务规则（决策表）
  - 数据字典
  - 非功能需求（量化）
  - 验收标准

交付时间:
  - PRD v1.0: 开发介入前 2 工作日
  - 回应问询: 1 工作日内
```

### 7.2 与测试的契约

```yaml
# 可测性契约

PRD 必须通过:
  - 所有需求点有验收标准
  - 异常场景 ≥ 80% 覆盖
  - 数据规则齐全
  - 性能指标量化

测试回传:
  - 可测性评审意见（评审会前 1 天）
  - 测试用例评审邀请
```

### 7.3 与运营的契约

```yaml
# 运营反馈契约

运营输出:
  - 月度用户行为报告
  - 客诉清单（按类型聚合）
  - A/B 测试结果

产品输出:
  - 季度迭代方向
  - 大版本功能预告
```

## 八、业务知识库构建

用 Markdown + Git + AI 构建业务团队的组织记忆：

```
实体类型:

[Project]  ──has──►  [Requirement]
[Requirement] ──realizes──► [Feature]
[Feature] ──has──► [UserStory]
[UserStory] ──has──► [AcceptanceCriteria]
[BusinessRule] ──governs──► [Feature]
[Customer] ──raised──► [Requirement]
```

**好处**：
- **需求可追溯**：每个功能对应哪个客户诉求
- **规则可继承**：同类产品的业务规则复用
- **经验可沉淀**：历史 CR 的决策依据保留

```
│                     └──────► 运营（反馈闭环）             │
└──────────────────────────────────────────────────────────┘
```

## 十二、名词解释

本篇出现的关键术语：

| 术语 | 定义 |
|------|------|
| **PRD** | Product Requirement Document,产品需求文档。业务侧最终交付给开发/测试的正式文档 |
| **Story Point** | 敏捷开发中对用户故事工作量的相对估算单位,常用 1/2/3/5/8/13 |
| **INVEST** | 好的用户故事应满足的 6 个特征：Independent / Negotiable / Valuable / Estimable / Small / Testable |
| **用户故事** | 用"As a... I want... So that..."格式描述的需求单元 |
| **验收标准（AC, Acceptance Criteria）** | 判断故事完成的客观标准,常用 Given-When-Then 格式 |
| **Given-When-Then** | BDD（行为驱动开发）的场景描述语法：给定前置条件,当触发动作,则预期结果 |
| **决策表** | 用表格形式枚举"条件组合 → 结果"的业务规则表达方式 |
| **状态机** | 描述对象生命周期中不同状态及其转换规则的模型 |
| **CR（Change Request）** | 变更请求单,对已批准 PRD 发起的变更申请 |
| **影响评估** | 对变更的功能/数据/接口/时间等多维度影响分析 |
| **Gate In / Gate Out** | 阶段的进入/退出条件,不满足不能流转 |
| **可测性（Testability）** | 需求能被测试用例清晰覆盖验证的程度 |
| **需求池（Backlog）** | 待排期的需求列表,按优先级动态管理 |
| **用户画像** | 对目标用户的抽象描述（年龄/职业/习惯/痛点等） |
| **故事地图** | 按"用户行为流"组织的故事可视化结构 |
| **访谈记录** | 与真实用户/客户对话的原始资料 |
| **竞品分析** | 对同类产品的功能/体验/商业模式对比 |
| **优先级（P0/P1/P2/P3）** | 对任务重要程度的分级 |
| **非目标（Non-Goal）** | 明确本期不做的事,避免范围蔓延 |
| **数据字典** | 对数据模型中每个字段的定义说明 |
| **A/B 测试** | 对比两个版本效果的实验方法 |
