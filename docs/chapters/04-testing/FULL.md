# 测试模块在企业项目生命周期中的定位与体系化建设

> 本文是**测试篇的统一长文版**,在原 Confluence 讨论（[存档](../../archive/original-discussion-confluence.md)）基础上补齐了**子页面设计 + 名词解释**,并与业务/开发/运维三篇对齐格式。

测试是研发全流程的"质量门闸"。守住门闸,产品才算可交付。本篇从三个层面展开：输入输出定义、Claude 工具化赋能、与多角色的协作规范。

---

## 一、测试模块的输入输出定义

测试在每个阶段都有明确的"契约"：

### 各阶段输入输出矩阵

| 阶段 | 输入（上游交付物） | 输出（测试交付物） | 上游角色 | 下游角色 |
|------|------------------|------------------|---------|---------|
| 需求评审 | PRD、用户故事、验收标准 | 需求可测性评审意见、测试范围初估 | 产品/业务 | 产品/开发 |
| 设计评审 | 技术方案、API 设计、数据模型 | 测试策略文档、风险评估 | 架构/开发 | 开发/项目经理 |
| 用例设计 | 已确认的需求+技术方案 | 测试用例、测试数据方案 | 产品/开发 | 开发（自测参考） |
| 开发联调 | 可部署的代码+部署说明 | 冒烟测试报告、环境验证结果 | 开发 | 测试自身 |
| 系统测试 | 通过冒烟的版本 | Bug 报告、测试执行报告 | 开发 | 开发（修复） |
| 回归/验收 | 修复后的版本 | 测试总结报告、上线准出意见 | 开发 | 项目经理/运维 |
| 线上运维 | 监控告警、用户反馈 | 线上问题复现报告、回归验证 | 运维/客服 | 开发 |

### 关键原则

**"没有合格的输入，就没有测试的启动"** — 测试守住质量的第一道防线。具体来说：

- **准入标准（Entry Criteria）**：需求必须有明确的验收标准；接口必须有契约文档；代码必须通过开发自测和 CI
- **准出标准（Exit Criteria）**：Bug 收敛趋势达标；核心用例 100% 通过；无 P0/P1 遗留

## 二、用 Claude 构建测试工具链

Claude 可以在测试的每个环节提供工具化支持：

### 1. 需求分析阶段 — 可测性评审

```
你是资深测试架构师。从"可测性"角度扫描 PRD,找出所有阻碍测试的点。

## 检查维度
1. 验收标准缺失或模糊
2. 边界条件未明确
3. 异常场景缺失
4. 数据规则不清
5. 非功能需求未量化
6. 权限粒度未说明
7. 状态流转不清
8. 异步/最终一致性缺说明

## 输出
| # | 章节 | 类型 | 问题 | 测试影响 | 建议补充 |

附加: 可测性评分 + 用例数初估 + 关键难点
```

### 2. 用例设计阶段 — 用例矩阵生成

```
基于需求 + 接口契约,生成结构化用例矩阵。

## 覆盖要求
- 正常流
- 参数校验
- 边界值
- 异常流
- 安全
- 并发 / 幂等性 / 性能基线 / 兼容性（按需）

## 输出
| 用例 ID | 模块 | 优先级 | 类型 | 前置条件 | 测试数据 | 步骤 | 预期结果 |

附加: 覆盖度评估 / 测试数据准备清单 / 自动化适配度
```

### 3. 接口变更阶段 — API diff

```
对比两版接口契约,识别变更。

## 输出
1. 变更清单（破坏性/行为/新增/废弃/移除）
2. 对测试影响（用例更新/新增/回归/脚本改动）
3. 对客户端影响
4. 兼容性建议（灰度/版本共存）
5. 回归 Checklist
```

### 4. 测试执行阶段 — 报告生成

```
根据执行数据,生成结构化测试报告。

## 结构
1. 摘要
2. 测试概述
3. 用例执行统计
4. Bug 统计（严重度/模块/趋势）
5. 性能结果
6. 风险评估
7. 质量评估（vs 准出标准）
8. 准出意见
9. 上线建议
10. 改进建议
```

### 5. Bug 分析阶段 — 汇总与根因

```
对一段时间的 Bug 做统计分析。

## 任务
1. 基础统计（数量/严重度/模块/状态）
2. 趋势分析（时间维度）
3. 模块热点（Top 5）
4. 根因模式（按类型：需求理解/边界/并发/性能/依赖/回归）
5. Bug 重开率分析
6. 线上逃逸 Bug 分析
7. 改进建议（技术/流程/文档）
```

## 三、与多角色的协作规范 — 生命周期串联

### 核心思路：用"契约"串联角色，用"门禁"保证质量

```
┌─────────┐    PRD + 验收标准     ┌─────────┐
│  业务/PM  │ ──────────────────► │         │
└─────────┘                      │         │
                                 │         │  测试策略
┌─────────┐    技术方案 + API 契约 │  测试    │ ──────────►  评审通过
│  开发/架构 │ ──────────────────► │         │
└─────────┘                      │         │
         ◄─── Bug 报告 ──────── │         │  准出报告
         修复版本 ──────────────► │         │ ──────────►
                                 └─────────┘             ┌─────────┐
                                                         │ 运维/发布 │
                                                         └─────────┘
```

### 具体规范建议

**1. 与业务/产品的契约**

| 项目 | 规范 |
|------|------|
| 输入格式 | PRD 必须包含验收标准（Given-When-Then 格式） |
| 评审机制 | 测试参与需求评审,输出可测性意见 |
| 变更管理 | 需求变更必须通知测试,重新评估影响范围 |

**2. 与开发的契约**

| 项目 | 规范 |
|------|------|
| 提测标准 | 开发自测通过 + CI 绿灯 + 部署文档 + 提测申请单 |
| 接口契约 | API 文档先行,变更需同步测试 |
| Bug 流转 | P0 4h / P1 当天 / P2 2 工作日 |
| 环境要求 | 测试环境由开发负责部署,测试负责验证 |

**3. 与运维的契约**

| 项目 | 规范 |
|------|------|
| 上线准出 | 测试签署准出报告后方可发布 |
| 灰度验证 | 测试提供灰度验证 checklist |
| 线上问题 | 测试协助复现,24h 内输出复现报告 |

### 用 Claude 落地这套规范的实际做法

**建立模板库**：
- 需求可测性评审 checklist 模板
- 测试策略文档模板
- 测试用例模板（完整 + 简版）
- Bug 报告模板
- 测试总结报告模板
- 准出评审报告模板
- 提测申请单模板
- API 契约 Markdown 模板（3 档详细度）
- API 变更记录模板

**建立 Skill**：
- `/test-review` — PRD 可测性分析（测试视角,更硬核）
- `/test-case-gen` — 根据需求+接口生成用例
- `/api-diff` — 接口变更影响分析
- `/test-report` — 测试执行数据 → 结构化报告
- `/bug-summary` — Bug 分布与趋势分析

## 四、总结：测试的核心定位

测试模块的本质是**质量信息的生产者和传递者**：

1. **向上游要求清晰的输入** — 不接受模糊需求、不接受没有自测的提测
2. **向下游提供可信的输出** — 报告要有数据支撑、结论要有风险量化
3. **用工具提升效率** — Claude 承担大量重复性的分析、生成、报告工作
4. **用规范保证一致性** — 模板化、流程化、门禁化

最终目标：**测试不是被动等待"被测对象"的角色，而是主动定义质量标准、驱动上下游协作的角色**。

## 五、测试生命周期的规范化运行框架

### 5.1 测试的"六阶段门禁模型"

```
 Gate In          阶段            Gate Out           下一阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 PRD已评审  ──►  ① 测试分析  ──►  测试策略已评审  ──►

 策略已批准  ──►  ② 用例设计  ──►  用例评审通过    ──►

 提测达标   ──►  ③ 测试执行  ──►  Bug收敛/用例完成 ──►

 Bug已修复  ──►  ④ 回归验证  ──►  回归通过        ──►

 全部通过   ──►  ⑤ 准出评审  ──►  准出报告签发    ──►

 已上线     ──►  ⑥ 线上验证  ──►  验证报告归档    ──►  Done
```

### 5.2 每个阶段的具体规范

**阶段①：测试分析**

```yaml
gate_in:
  必须项:
    - PRD 文档（含验收标准）
    - 用户故事地图 / 业务流程图

deliverables:
  - 测试策略文档（范围/类型/风险/排期/环境）
  - 需求可测性反馈

gate_out:
  - 测试策略评审通过
  - 需求疑问全部关闭
```

**阶段②：用例设计**

```yaml
deliverables:
  - 测试用例集（核心流 100% / 异常 ≥ 80%）
  - 测试数据方案
  - 自动化可行性分析

gate_out:
  - 用例评审通过（产品 + 开发参与）
  - 测试数据准备就绪
```

**阶段③：测试执行**

```yaml
gate_in（提测达标）:  # ← 测试守住质量的关键门禁
  必须项:
    - 提测申请单
    - CI 流水线绿灯
    - 单测覆盖率 ≥ 80%
    - 冒烟通过
    - 接口文档与代码一致
    - 测试环境可访问

deliverables:
  - 每日测试进度
  - Bug 报告
  - 测试执行报告

gate_out:
  - 核心用例通过率 100%
  - 无 P0/P1 遗留
  - Bug 收敛趋势达标
```

**阶段④：回归验证**

```yaml
gate_in:
  - Bug 修复完成并部署

deliverables:
  - 回归测试报告
  - 新增 Bug（如有）

gate_out:
  - 所有已修复 Bug 验证通过
  - 回归用例 100% 通过
```

**阶段⑤：准出评审**

```yaml
deliverables:
  - 准出报告（质量数据 + 风险 + 结论）
  - 上线 Checklist
  - 回滚方案

gate_out:
  - 测试负责人签字
  - 项目经理批准
  - 运维接收确认
```

**阶段⑥：线上验证**

```yaml
deliverables:
  - 线上验证报告
  - 线上问题复现报告（如有）
  - 测试回顾与改进项

gate_out:
  - 灰度验证通过
  - 监控指标正常
  - 报告归档
```

## 七、跨角色协作的"契约模板"

### 7.1 与产品经理的契约

```markdown
## 产品 → 测试 交付契约

### 交付物要求
1. **需求文档**
   - 格式：Confluence / Wiki / Markdown
   - 必须包含：用户故事（As a... I want... So that...）
   - 必须包含：验收标准（Given... When... Then...）
   - 必须包含：业务规则表（条件 → 结果）

2. **交付时间**
   - 需求评审前 2 工作日提供完整 PRD
   - 评审后 1 工作日内关闭测试疑问

### 测试 → 产品 交付物
1. 需求可测性评审意见（评审会前 1 天）
2. 测试用例评审邀请（用例完成后）
3. 测试进度周报（每周五）
4. 准出报告（测试完成后）
```

### 7.2 与开发的契约

```markdown
## 开发 → 测试 交付契约

### 提测标准（不满足任一项则拒绝提测）
1. □ 代码已合并到测试分支
2. □ CI 流水线通过（单测覆盖率 ≥ 80%）
3. □ 提测申请单完整
4. □ 接口文档已更新且与代码一致
5. □ 测试环境已部署且可访问

### Bug 流转规范
- 开发收到 Bug 后 4h 内确认
- P0: 2h 内开始修复
- P1: 当天内修复
- P2: 2 个工作日内修复
- 修复后在 Issue 标注修复版本,@测试回归
```

### 7.3 与运维的契约

```markdown
## 测试 → 运维 交付物

### 上线准出报告
- 版本、测试周期、质量数据
- Bug 统计（按严重级别）
- 风险评估 + 缓解措施
- 准出意见（通过 / 有条件 / 不通过）
- 签署人

### 灰度验证 Checklist
- 核心业务流程验证
- 数据库数据一致性验证
- 接口响应时间基线对比
- 错误日志监控（30 min）
- 关键指标监控（成功率、P99）
```

## 八、用 Knowledge Graph 构建测试知识库

用 Markdown + Git + AI 构建测试团队的组织记忆：

```
实体类型:

[Project]   ──has_version──►   [Version]
[Version]   ──tested_by──►    [TestCycle]
[TestCycle] ──found──►        [Bug]
[TestCycle] ──produced──►     [TestReport]
[Bug]       ──affects──►      [Module]
[Module]    ──owned_by──►     [Developer]
[TestCase]  ──covers──►       [Requirement]
```

**好处**：
- **历史可追溯**：某模块历史出过多少 Bug
- **风险可量化**：哪个开发的提测质量最差
- **经验可复用**：类似需求上次的测试策略

## 十二、名词解释

| 术语 | 定义 |
|------|------|
| **Gate In / Gate Out** | 阶段的进入/退出条件,不满足不能流转 |
| **Entry Criteria / Exit Criteria** | 准入标准 / 准出标准,对应 Gate In / Gate Out |
| **提测达标** | 开发交付给测试的版本必须满足的硬性条件 |
| **准出报告** | 测试完成后对能否上线的正式意见书 |
| **冒烟测试** | 核心流程快速验证,确保基本功能可用 |
| **回归测试** | 修改后对原有功能的重新验证 |
| **用例** | Test Case,测试步骤 + 预期结果的标准化描述 |
| **用例覆盖率** | 被用例覆盖的需求/代码比例 |
| **等价类划分** | 把输入域分成若干子类,每类取代表测试 |
| **边界值** | 输入域的上下限（空、null、最大、最小、临界） |
| **决策表** | 表格化枚举"条件组合 → 结果" |
| **AAA 模式** | Arrange-Act-Assert,测试标准结构 |
| **Given-When-Then** | BDD 场景描述语法 |
| **BDD** | Behavior Driven Development,行为驱动开发 |
| **TDD** | Test Driven Development,测试驱动开发 |
| **自动化测试** | 用脚本代替人工执行的测试 |
| **接口测试** | 对 API 的测试（常用 Postman / Tavern） |
| **性能测试** | QPS、延迟、并发等非功能测试（常用 JMeter / Locust） |
| **安全测试** | SQL 注入、XSS、越权等攻击向量测试 |
| **兼容性测试** | 浏览器、OS、设备等环境组合测试 |
| **P0/P1/P2/P3** | Bug 严重级别（阻塞 / 严重 / 一般 / 轻微） |
| **Bug 收敛** | 发现速度逐步降低,修复速度超过发现速度 |
| **Bug 重开率** | 复测发现修复不完整的比例 |
| **线上逃逸 Bug** | 测试阶段未发现,上线后暴露的 Bug |
| **准入标准** | 测试开始前必须满足的条件 |
| **准出标准** | 测试结束前必须满足的条件 |
| **灰度验证** | 灰度发布阶段的测试动作 |
| **可测性（Testability）** | 需求能被测试清晰覆盖验证的程度 |
| **测试数据** | 测试执行所需的输入数据 |
| **测试环境** | 与生产独立的测试用环境 |
| **Mock** | 测试替身,模拟外部依赖 |
| **测试策略** | 对本版本测试范围、方法、资源的总体规划 |
| **测试计划** | 测试执行的排期和分工 |
| **API 契约** | 接口规范,本项目推荐 Markdown 格式替代 Swagger |
| **接口变更** | API 的字段/行为修改 |
| **破坏性变更** | Breaking Change,不向下兼容的变更 |
