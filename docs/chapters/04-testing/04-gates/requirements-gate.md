# 需求评审门禁

## Gate In（进入需求评审的条件）

- [ ] PRD 文档已提交
- [ ] 业务流程图已提供
- [ ] 验收标准已编写（Given-When-Then 格式）
- [ ] 相关方均已收到评审邀请

## Gate Out（需求评审通过的标准）

### 可测性检查（测试负责）

- [ ] 每个功能点都有明确验收标准
- [ ] 异常场景已考虑
- [ ] 边界条件已明确
- [ ] 数据规则清晰（必填/选填、长度、格式）
- [ ] 性能/安全/兼容性需求明确

### 可实现性检查（开发负责）

- [ ] 技术可行性确认
- [ ] 依赖服务已评估
- [ ] 工期估算完成

### 完整性检查（产品负责）

- [ ] 用户故事完整
- [ ] 业务闭环清晰
- [ ] 和现有功能的关系说明

## 不通过的处理

| 问题 | 处理 |
|------|------|
| 验收标准模糊 | 产品补齐后重评 |
| 异常场景缺失 | 产品补充后重评 |
| 技术不可行 | 开发提出，产品调整方案 |

## 评审产物

评审通过后，测试必须输出：
- 需求可测性评审结论
- 初步测试范围评估
- 风险预警（如有）
