# 第 04 篇 · 测试篇

> **状态**: ✅ 骨架完成（四篇中最成熟）
> **目标读者**: 测试工程师 / QA / 测试负责人

## 一、本篇目标

把"测试"作为企业项目全流程中的一个**独立、可管理、可度量**的模块来建设，解决：

1. **输入输出规范化** — 每个阶段交付什么、接收什么
2. **角色协作标准化** — 与业务/开发/运维如何对接
3. **门禁把关常态化** — 不合格就退回，不迁就
4. **度量可视化** — 质量用数据说话

## 四、核心理念

### 「没有合格的输入，就没有合格的输出」

测试守住质量的第一道防线 = **不接收不合格的输入**。

- 需求模糊？退回业务
- 自测没做？退回开发
- 环境起不来？退回开发
- 接口文档不一致？退回开发

### 「契约先行」

- 与业务的契约：[`product-contract.md`](./03-roles-contracts/product-contract.md)
- 与开发的契约：[`dev-contract.md`](./03-roles-contracts/dev-contract.md)
- 与运维的契约：[`ops-contract.md`](./03-roles-contracts/ops-contract.md)

### 「度量驱动」

用数据说话，不用感觉说话。核心看板：
- 提测一次通过率 ≥ 80%
- 核心用例通过率 = 100%
- 线上缺陷逃逸率 ≤ 3%
