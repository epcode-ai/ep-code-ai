# 运维模块在企业项目生命周期中的定位与体系化建设

运维是"让代码真正服务用户"的最后一公里。线上稳定靠运维,出事能救也靠运维。本篇从三个层面展开：输入输出定义、Claude 工具化赋能、与多角色的协作规范。

---

## 一、运维模块的输入输出定义

运维在每个阶段都有明确的"契约"：

### 各阶段输入输出矩阵

| 阶段 | 输入（上游交付物） | 输出（运维交付物） | 上游角色 | 下游角色 |
|------|------------------|------------------|---------|---------|
| 上线接收 | 准出报告、可发布的版本 | 接收确认、发布计划 | 测试 | 运维内部 |
| 发布执行 | 发布计划、灰度策略 | 上线报告、发布记录 | 运维 | 业务/用户 |
| 日常监控 | 业务、应用、基础层指标 | 告警、大盘、趋势 | 全系统 | 运维 |
| 告警响应 | 告警事件 | 处置动作、Runbook 执行 | 监控 | 开发/业务（通知） |
| 故障处理 | SEV 事件 | 故障报告、状态通报 | 监控/用户反馈 | 全员 |
| 故障复盘 | 故障报告、时间线 | 复盘报告、改进项 | 运维 | 全员 |
| 容量规划 | 业务增长预测、历史数据 | 扩容计划、成本预算 | 业务/运维 | 采购/预算 |
| 值班交接 | 上班期事件 | 交接单 | 前值班 | 后值班 |
| 线上反馈 | 监控趋势、用户反馈 | 业务改进建议 | 运维 | 业务/开发 |

### 关键原则

**"先恢复业务，后定位根因"** + **"复盘不追责，改进才有用"** — 运维侧的两条黄金法则。

- **准入标准（Entry Criteria）**：准出报告签字;发布窗口已约;回滚方案就绪;值班到岗
- **准出标准（Exit Criteria）**：灰度验证通过;监控指标稳定;无异常告警持续观察 24h

## 二、用 Claude 构建运维工具链

Claude 在运维侧最有价值的点是"**减少人肉分析 + 缩短 MTTR**"：

### 1. 日志分析 — 海量日志找规律

```
分析以下生产日志,识别异常模式。

## 日志内容
[粘贴 100-2000 行]

## 任务
1. 聚类错误（按模式,忽略 ID）
2. 突增点识别
3. 错误链分析
4. 与历史故障匹配
5. 严重度评估

## 输出
- 异常模式 Top N
- 时间线
- 处置建议优先级
- 需要补充的监控指标
```

### 2. 告警归因 — 风暴聚合

```
以下是最近触发的所有告警,帮我聚合和归因。

## 告警列表
| 时间 | 告警名 | 服务 | 值 | 级别 |

## 服务拓扑
[服务依赖关系]

## 最近变更
[1 小时内的发布/配置]

## 输出
1. 相关告警聚合
2. 推断根因（top 3 概率）
3. 每个根因的验证方法
4. 处置优先级
5. 告警规则改进建议
```

### 3. 故障诊断 — 多轮对话副驾驶

```
我正在处理生产故障,你做我的副驾驶。我们多轮对话。

## 故障现象
[告警 + 用户反馈]

## 已知信息
- 服务拓扑
- 最近变更
- 当前监控数据
- 已尝试过的

## 请给我
1. 接下来该查的 3-5 个方向（按价值排序）
2. 每个方向的具体命令/查询
3. 每个可能结果对应的下一步

我会边查边告诉你结果,我们迭代。
```

### 4. Runbook 生成 — 从历史经验

```
基于以下信息,帮我生成一份 Runbook。

## 场景信息
- 告警名 + 规则
- 典型现象 / 用户感知
- 历史故障摘要
- 已知处置经验
- 服务上下文

## 结构要求
1. 故障现象
2. 严重度评估
3. 快速定位（5 分钟内）
4. 处置方案（至少 3 个,按条件区分）
5. 常见根因（按频率）
6. 升级路径
7. 事后必做

## 要求
- 具体命令可执行
- 每个动作"如果没改善怎么办"
- 需经值班人员评审才正式使用
```

### 5. 复盘报告初稿 — 从故障群聊

```
基于以下信息生成故障复盘报告初稿。

## 原则
Blameless Culture（对事不对人）。

## 输入
- 故障群时间线
- 监控数据
- 处置动作
- 初步原因

## 结构
1. 摘要
2. 时间线（精确到分钟）
3. 故障经过
4. 5 Why 根因分析
5. 做得好 / 做得不好
6. 改进项（SMART 原则）
7. 教训

## 要求
- 不指名道姓
- 5 Why 挖到流程/文化层
- 改进项具体可执行
- 标注"AI 生成初稿,需人工核对"
```

### 6. 容量预测 — 扩容规划

```
基于以下历史数据,预测未来 N 个月容量需求。

## 历史数据
QPS / DAU / 资源用量（近 6-12 月）

## 业务计划

- 新功能上线

## 当前资源

## 输出
1. 预测结果（按月）
2. 资源需求（扩容量 + 时机）
3. 扩容节奏建议
4. 成本分析
5. 风险与应对
```

## 三、与多角色的协作规范 — 生命周期串联

### 核心思路：用"契约"串联角色，用"演练"保证稳定

```
┌─────────┐   准出报告       ┌─────────┐
│ 测试/QA  │ ──────────────► │         │
└─────────┘                  │         │  运行指标 / 故障报告
                             │ 运维/SRE │ ──────────► 业务 / 管理层
┌─────────┐   代码 / 部署文档│         │
│  开发    │ ──────────────► │         │  线上反馈
└─────────┘                  │         │ ──────────► 业务 / 开发
         ◄── 故障协助 ──── │         │
         ◄── 线上反馈 ──── │         │
                             └─────────┘
```

### 具体规范建议

**1. 与测试的契约**

| 项目 | 规范 |
|------|------|
| 上线接收 | 准出报告签字后才接受版本 |
| 灰度验证 | 测试提供灰度 Checklist,运维逐条验证 |
| 回滚协同 | 回滚方案必须可演练,演练后才能真发版 |

**2. 与开发的契约**

| 项目 | 规范 |
|------|------|
| 部署文档 | 发版前提交,包含所有环境变化 |
| 监控埋点 | 发版前对齐,不事后补 |
| 故障协助 | P0 5 分钟响应;P1 15 分钟响应 |
| 技术债反馈 | 每季度向开发/架构同步线上问题 |

**3. 与业务的契约**

| 项目 | 规范 |
|------|------|
| 运行数据 | 月度/季度提供用户行为 + 故障数据 |
| 故障通报 | SEV-1/2 事件 24h 内对业务侧同步 |
| 容量沟通 | 大促/活动前 2 周确认容量需求 |

**4. 与安全的契约**

| 项目 | 规范 |
|------|------|
| 漏洞响应 | CVE 24h 内评估,高危 72h 内修复 |
| 审计日志 | 敏感操作（DB/配置/权限）必须留痕 |
| 合规 | 数据留存、脱敏、访问控制定期评审 |

## 四、总结：运维的核心定位

运维模块的本质是**让系统稳定运行 + 出事能救**：

1. **向上游要求可运维**：部署文档齐备、监控埋点对齐、回滚可演练
2. **向下游提供稳定**：SLA 达标、告警及时、故障可恢复
3. **事后沉淀经验**：每次故障都要 Runbook 和复盘
4. **数据驱动改进**：MTTR 要降、故障率要降、容量要前瞻

最终目标：**运维不是"消防队",而是让系统从"能跑"到"能稳定跑,出事能救"的专业团队**。

## 五、运维生命周期的规范化运行框架

### 5.1 运维的"七阶段门禁模型"

```
 Gate In              阶段              Gate Out            下一阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

 准出签字      ──►  ① 上线接收    ──►  接收确认 + 发布计划   ──►

 发布计划就绪  ──►  ② 发布执行    ──►  上线完成 + 灰度通过   ──►

 已上线       ──►  ③ 日常监控    ──►  稳定运行               ──►

 监控告警     ──►  ④ 告警响应    ──►  按 Runbook 处置        ──►

 严重事件     ──►  ⑤ 故障处理    ──►  业务恢复 + 故障报告    ──►

 故障恢复     ──►  ⑥ 故障复盘    ──►  改进项跟踪             ──►

 周期到       ──►  ⑦ 容量 / 值班 ──►  扩容计划 / 交接单      ──►  Done
```

### 5.2 每个阶段的具体规范

**阶段①：上线接收**

```yaml
gate_in:
  必须项:
    - 准出报告（测试签字）
    - 发布计划草稿
    - 部署文档 + 回滚方案
    - 数据库 migration + rollback
    - 监控埋点确认

deliverables:
  - 接收确认签字
  - 发布计划 final
  - 应急联系人清单

gate_out:
  - 发布窗口已预约
  - 全员已知悉时间和角色
```

**阶段②：发布执行**

```yaml
trigger: 发布窗口到达

流程:
  T-1h: 团队集合 + 监控开启
  T+0: 执行部署（先 1 实例）
  T+5m: 冒烟测试
  T+15m: 灰度 1% → 5%
  T+1h: 20% → 50%
  T+4h: 100%
  T+24h: 稳定观察 + 通知发布完成

回滚触发:
  - 错误率 > 2%
  - 核心业务不可用
  - 数据错误

deliverables:
  - 发布报告（T+24h 内）
```

**阶段③：日常监控**

```yaml
三层监控:
  - 业务层: 核心转化率、关键流程时长
  - 应用层: RED（Rate / Error / Duration）
  - 基础层: CPU / 内存 / 磁盘 / 网络

报告节奏:
  - 日: 值班观察
  - 周: 周报 + 趋势
  - 月: 月度故障 + 容量
  - 季: 架构级复盘
```

**阶段④：告警响应**

```yaml
级别:
  - P0 紧急: 5 分钟响应,电话/短信
  - P1 严重: 15 分钟响应,IM/短信
  - P2 一般: 1 小时响应,IM
  - P3 轻微: 下一工作日

处置流程:
  1. 确认真实性
  2. 按 Runbook 定位
  3. 处置（先恢复）
  4. 验证
  5. 事后总结（更新 Runbook）
```

**阶段⑤：故障处理**

```yaml
角色:
  - IC（Incident Commander）: 统筹,唯一决策,不动手
  - Tech Lead: 技术诊断
  - Comms Lead: 对内对外沟通

黄金 5 问:
  1. 什么变了?
  2. 什么还能工作?
  3. 什么在报错?
  4. 已知的类似问题?
  5. 如果不是代码问题,是什么?

处置原则:
  先恢复业务,后定位根因

deliverables:
  - 故障报告（24h 内）
```

**阶段⑥：故障复盘**

```yaml
时机: 故障恢复后 24-48h

原则:
  - Blameless（对事不对人）
  - 5 Why 挖到流程/文化层
  - 改进项 SMART

产出:
  - 复盘报告
  - 改进项清单（按责任人、截止日期）
  - 跟踪机制（周例会 review 进展）
```

**阶段⑦：容量 + 值班**

```yaml
容量规划:
  - 季度评估增长
  - 大促前 2 周确认
  - 扩容演练

值班制度:
  - 轮值表
  - 交接单
  - 工具和权限
```

## 八、运维知识库构建

用 Markdown + Git + AI 构建运维团队的组织记忆：

```
实体类型:

[Service]   ──has_runbook──►   [Runbook]
[Service]   ──depends_on──►   [Service]
[Alert]     ──triggers──►      [Runbook]
[Incident]  ──resolved_by──►   [Runbook]
[Incident]  ──causes──►        [Postmortem]
[Postmortem] ──produces──►     [ImprovementAction]
[ImprovementAction] ──owner──► [Person]
```

**好处**：
- **故障可追溯**：某服务历史上出过多少事
- **Runbook 复用**：同类告警用同一手册
- **改进项跟踪**：复盘不是一阵风,真的改到位
- **容量历史**：下次大促有基线

## 十、总结：运维负责人的行动清单

| 优先级 | 行动项 | 预期收益 |
|--------|--------|---------|
| **P0** | 至少 5 份核心 Runbook | 值班人不再"凭经验" |
| **P0** | 建立值班 SOP + 故障响应流程 | 出事有章可循 |
| **P0** | Blameless 复盘机制 | 团队敢报问题,改进可持续 |
| **P1** | 三层监控体系（业务/应用/基础） | 异常不漏报 |
| **P1** | 告警降噪 + 分级 | 真告警不被淹没 |
| **P1** | 配置中心 + 多环境审计 | 配置漂移可防 |
| **P2** | 定期混沌演练 | 韧性提前暴露 |
| **P2** | 容量规划数据化 | 大促不手忙脚乱 |
| **P3** | AI 辅助日志 / 告警 / 复盘 | 缩短 MTTR |

## 十二、名词解释

| 术语 | 定义 |
|------|------|
| **SLA** | Service Level Agreement,服务等级协议（对用户承诺） |
| **SLO** | Service Level Objective,服务等级目标（内部目标） |
| **SLI** | Service Level Indicator,服务等级指标（实际数据） |
| **SRE** | Site Reliability Engineering,Google 提出的运维工程范式 |
| **MTTD** | Mean Time To Detect,平均发现时间 |
| **MTTR** | Mean Time To Recover,平均恢复时间 |
| **MTBF** | Mean Time Between Failures,故障间隔时间 |
| **SEV** | Severity,故障严重度（SEV-1 ~ SEV-4） |
| **IC** | Incident Commander,故障指挥官（只决策不动手） |
| **Runbook** | 故障处置手册,针对特定告警的标准操作流程 |
| **Postmortem** | 故障复盘,恢复后的深度分析文档 |
| **Blameless** | 无指责文化,复盘对事不对人 |
| **5 Why** | 连续问 5 次为什么,挖到根因的方法 |
| **RED 方法** | 对服务的 Rate（速率）/ Errors（错误）/ Duration（延迟）三指标监控 |
| **USE 方法** | 对资源的 Utilization（使用率）/ Saturation（饱和度）/ Errors（错误）监控 |
| **Alertmanager** | Prometheus 生态的告警管理组件 |
| **Webhook** | HTTP 回调接口,告警转发到 IM 常用 |
| **灰度发布** | 按比例或范围逐步放量的发布方式 |
| **蓝绿发布** | 两套环境切换的发布方式 |
| **滚动发布** | 实例逐个替换的发布方式 |
| **回滚** | 版本回到上一稳定状态 |
| **降级** | 关闭非核心功能保核心功能 |
| **限流** | 限制请求速率保护系统 |
| **熔断** | 调用失败达阈值自动断开 |
| **混沌工程** | 主动注入故障验证系统韧性 |
| **Chaos Mesh** | 云原生混沌工程工具 |
| **告警风暴** | 短时间大量相关告警同时触发 |
| **级联告警** | 一个根因引发的多个告警 |
| **配置漂移** | 不同环境配置不一致 |
| **SBOM** | Software Bill of Materials,软件物料清单 |
| **CVE** | Common Vulnerabilities and Exposures,公开漏洞编号 |
| **容量规划** | 基于历史数据和预期增长预估资源 |
| **爆炸半径** | 混沌实验可能影响的范围 |
