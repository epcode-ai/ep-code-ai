## 3. 灰度策略

### 3.1 灰度方案

- 灰度维度: [按百分比 / 按用户 / 按地域]
- 灰度节奏:

| 阶段 | 时间 | 流量 | 验证项 |
|------|------|------|-------|
| 阶段 1 | T+0:00 | 1% | 基础健康检查 |
| 阶段 2 | T+0:30 | 5% | 业务指标 |
| 阶段 3 | T+1:00 | 20% | 全面监控 |
| 阶段 4 | T+2:00 | 50% | 对比基线 |
| 阶段 5 | T+4:00 | 100% | 持续观察 |

### 3.2 灰度观察点

| 指标 | 基线 | 告警阈值 |
|------|------|---------|
| API 错误率 | 0.1% | > 0.5% |
| P99 延迟 | 200ms | +20% |
| CPU | 50% | > 80% |
| 内存 | 60% | > 80% |
| 业务核心转化率 | - | -5% |

## 4. 回滚方案

### 4.1 触发条件

立即回滚（任一满足）：
- [ ] 核心业务功能不可用
- [ ] 错误率飙升 > 2%
- [ ] 数据错误/丢失
- [ ] 安全漏洞暴露
- [ ] 性能严重劣化（P99 +50%）

### 4.2 回滚步骤

```bash
# Step 1: 切换流量
kubectl rollout undo deployment/xxx

# Step 2: 验证
curl http://service/health

# Step 3: 数据回滚（如需要）
./scripts/rollback-migration.sh
```

### 4.3 回滚后跟进

- [ ] 保留故障现场（日志/监控截图）
- [ ] 通知相关方
- [ ] 24 小时内产出故障报告
- [ ] 计划下次发布时间

## 5. 发布流程（时间线）

```
T-24h   ┌─ 最终 go/no-go 会议
        └─ 所有前置条件检查

T-60min ┌─ 团队 standby
        ├─ 开启监控放大镜
        └─ 备份当前版本

T-30min ┌─ 执行预发布（pre-deploy）
        └─ 烟测通过

T-0     ┌─ 开始正式发布
        └─ 部署到生产（1 个实例）

T+5min  ┌─ 健康检查
        └─ 灰度 1%

T+15min → 灰度 5%（按 3.1 节奏推进）
...
T+4h    → 灰度 100%

T+24h   ┌─ 稳定观察
        ├─ 通知发布完成
        └─ 归档发布报告

T+1w    └─ 发布复盘
```
